# Сравнение с .NET

Предыдущие главы разбирали инструменты Go по отдельности. Эта глава сводит всё вместе и смотрит на тулинг как на **систему** — в сравнении с тем, к чему вы привыкли в .NET. Главный тезис: и .NET, и Go дают **единый официальный тулчейн** через одну команду-обёртку (`dotnet` и `go`), но философия наполнения у них разная. В .NET значительная часть тяжёлого тулинга (профайлеры, продвинутый отладчик) — это отдельные, часто коммерческие продукты вокруг платформы. В Go максимум втянут в сам язык: профилирование и трассировка встроены, форматирование каноническое, и лишь отладчик вынесен в отдельный (но единственный) инструмент.

## Единый CLI: `dotnet` против `go`

В обоих мирах повседневная работа идёт через одну команду с подкомандами — и это огромный комфорт по сравнению с разрозненными утилитами. Соответствие почти прямое:

| Действие | .NET CLI | Go CLI |
| --- | --- | --- |
| Собрать | `dotnet build` | `go build` |
| Запустить | `dotnet run` | `go run` |
| Прогнать тесты | `dotnet test` | `go test` |
| Подготовить артефакт | `dotnet publish` | `go build` (даёт один статический бинарник) |
| Установить CLI-утилиту | `dotnet tool install -g` | `go install pkg@version` |
| Добавить зависимость | `dotnet add package` | `go get pkg@version` |

Главное различие — в природе артефакта. `dotnet publish` готовит развёртывание (зависит от модели: framework-dependent или self-contained, с рантаймом или без). `go build` по умолчанию выдаёт **один статически слинкованный бинарник без внешних зависимостей** — его можно просто скопировать на целевую машину или положить в `scratch`-контейнер. Отдельной команды `publish` в Go нет, потому что `build` уже даёт готовый к раздаче результат. Сравнение моделей сборки и развёртывания детальнее — в разделах про структуру проекта и деплой; здесь нам важно лишь, что оба тулчейна **единые и официальные**, без зоопарка сторонних сборщиков.

> **Параллель с .NET:** концептуально `go` — это «`dotnet` для Go»: одна точка входа на сборку, запуск, тесты, зависимости. Психологически переход лёгкий — вы меняете не парадигму, а имена подкоманд. Самый заметный сдвиг — отсутствие `.sln`/`.csproj` как описания проекта (его роль играют `go.mod` + соглашения о каталогах) и единый бинарник вместо набора DLL.

## Профилирование: встроенное против внешнего

Здесь — самое яркое различие и, пожалуй, главная причина написать эту главу.

В **.NET** для профилирования вы выходите *за пределы* SDK к специализированным инструментам:
- **dotTrace** (JetBrains) — CPU/timeline-профайлер, коммерческий.
- **dotMemory** (JetBrains) — профайлер памяти, коммерческий.
- **Профайлер Visual Studio** — CPU Usage, Memory Usage, Allocation (в составе VS).
- **PerfView** (Microsoft, бесплатный) — мощный, на событиях ETW, но с крутой кривой обучения.
- **`dotnet-trace`/`dotnet-counters`/`dotnet-dump`** — кроссплатформенные диагностические CLI-утилиты, подключаемые к процессу по PID.

В **Go** профилирование — это `pprof`, **часть рантайма и стандартной библиотеки** (см. [главу 3](./03-profiling-pprof.md)). Ничего коммерческого, ничего стороннего: импортировали `_ "net/http/pprof"` — и у живого сервиса есть HTTP-эндпоинты с CPU/heap/goroutine-профилями. Анализ — встроенным `go tool pprof` (включая flame graphs). Для трассировки выполнения — `go tool trace`, тоже часть тулчейна.

Практический эффект этого различия огромен: **профилировать прод в Go — рутинная операция**. Открыть `/debug/pprof/profile` на проблемном поде и за минуту получить flame graph — это нормальная практика. В .NET профилирование прода исторически было более тяжёлым мероприятием (хотя `dotnet-trace`/`dotnet-counters` и diagnostics в современном .NET этот разрыв заметно сократили).

> **Параллель с .NET:** если вы привыкли к богатому GUI dotTrace/dotMemory — будьте готовы, что pprof проще и аскетичнее по интерфейсу, но зато всегда под рукой и бесплатен. Это очень в духе Go: не самый красивый инструмент, но встроенный, достаточный и одинаковый у всех. Метрики `B/op`/`allocs/op` из `go test -benchmem` закрывают ту же нишу, что `[MemoryDiagnoser]` в BenchmarkDotNet.

## Отладка: интегрированная против отдельного инструмента

| | .NET | Go |
| --- | --- | --- |
| Движок отладки | `vsdbg` (часть VS/VS Code), SOS/WinDbg | **Delve** (`dlv`) — отдельный CLI |
| Поставка | вместе с IDE/SDK | ставится отдельно (`go install`) |
| Понимание рантайма | CLR, `async`-стейт-машины, GC-куча | горутины, каналы, динамические стеки |
| GUI | *является* реализацией (VS) | VS Code и GoLand — фронтенды над Delve |
| Удалённая отладка | Remote Debugger / vsdbg в контейнере | `dlv --headless` |

Ключевая разница описана в [главе 2](./02-debugging-delve.md): в .NET графический отладчик VS *и есть* движок, отдельного «общего ядра для всех IDE» нет. В Go есть один движок — **Delve**, — а редакторы лишь рисуют над ним GUI. Поэтому в Go полезно уметь работать с `dlv` из терминала: этот навык переносится между любыми редакторами и незаменим для контейнеров (headless-режим). И там, и там отладчик специально знает свой рантайм — gdb для Go так же бесполезен, как нативный отладчик без SOS для CLR.

## Форматирование и анализ

Вот где философии расходятся сильнее всего — в отношении к **стилю и статическому анализу**.

| Задача | .NET | Go |
| --- | --- | --- |
| Форматирование | `dotnet format` (по `.editorconfig`) | `gofmt` / `go fmt` (**канон, без настроек**) |
| Базовый анализ багов | Roslyn-анализаторы (CA-правила) | `go vet` (встроен в тулчейн) |
| Строгий анализ | StyleCop, SonarAnalyzer, сторонние | `staticcheck` (стандарт сообщества) |
| Конфигурируемость стиля | высокая (`.editorconfig`) | **нулевая** (один стиль на всех) |

Фундаментальное различие — в **конфигурируемости**. `dotnet format` подчиняется вашему `.editorconfig`: команда сама решает про отступы, скобки, порядок `using`. В Go форматирование задаётся `gofmt` и **не настраивается вообще** — ни табы-vs-пробелы, ни стиль скобок не обсуждаются, потому что у них нет опций. Это сознательное решение языка: убрать целый класс споров и сделать любой Go-код в мире визуально одинаковым. Для пришедшего из .NET это поначалу непривычно («а где мои настройки?»), но быстро воспринимается как облегчение — стиль просто перестаёт быть темой code review.

По анализу соответствие чище: `go vet` ≈ встроенные Roslyn-анализаторы (детектор вероятных багов, а не стиля), а `staticcheck` ≈ более строгим сторонним анализаторам. В реальных проектах поверх ставят агрегатор `golangci-lint`, запускающий десятки линтеров разом, — это и есть Go-аналог «включить расширенный набор анализаторов» в .NET.

> **Параллель с .NET:** представьте, что `.editorconfig` в вашей команде *запрещён*, а формат задан единым неизменяемым стандартом для всей экосистемы языка — это и есть `gofmt`. Анализаторы при этом остаются (`go vet` + `staticcheck`/`golangci-lint` ≈ ваш привычный набор Roslyn/StyleCop/Sonar), но война за стиль форматирования просто отменена на уровне языка.

## Сводная таблица соответствий

Финальный «ориентир» по всему тулингу раздела:

| Задача | .NET | Go |
| --- | --- | --- |
| Сборка / запуск / тесты | `dotnet build`/`run`/`test` | `go build`/`run`/`test` |
| Артефакт деплоя | `dotnet publish` (DLL + рантайм) | `go build` (один статический бинарник) |
| Добавить зависимость | `dotnet add package` (NuGet) | `go get pkg@version` |
| Глобальная CLI-утилита | `dotnet tool install -g` | `go install pkg@version` |
| Инструмент проекта | local tool + `dotnet-tools.json` | директива `tool` в `go.mod` + `go tool` (1.24) |
| Кэш пакетов | `~/.nuget/packages` | `GOPATH/pkg/mod` |
| Отладчик | `vsdbg` (в VS/VS Code) | Delve (`dlv`) — отдельный CLI |
| Удалённая отладка | Remote Debugger / vsdbg в контейнере | `dlv --headless --listen` |
| CPU-профиль | dotTrace / VS Profiler | `pprof` (`/debug/pprof/profile`) |
| Профиль памяти | dotMemory / VS | `pprof` (`/debug/pprof/heap`) |
| Анализ конкурентности | concurrency-профайлеры / PerfView | `pprof` (goroutine/block/mutex) |
| Трассировка выполнения | PerfView / `dotnet-trace` (ETW) | `go tool trace` |
| Диагностика живого процесса | `dotnet-counters`/`dotnet-dump` (по PID) | эндпоинты `net/http/pprof` (по HTTP) |
| Бенчмарки | BenchmarkDotNet (`[MemoryDiagnoser]`) | `go test -bench -benchmem` |
| Форматирование | `dotnet format` (по `.editorconfig`) | `gofmt` / `go fmt` (канон) |
| Базовый анализ | Roslyn-анализаторы | `go vet` |
| Строгий анализ | StyleCop / Sonar | `staticcheck` / `golangci-lint` |

## Итог

- **Единый CLI** в обоих мирах: `dotnet` ↔ `go` — одна точка входа на сборку, запуск, тесты, зависимости. Главное отличие артефакта: `go build` даёт **один статический бинарник**, поэтому отдельного `publish` в Go нет.
- **Профилирование** — самое яркое расхождение: в .NET это внешние, часто коммерческие инструменты (dotTrace/dotMemory/VS Profiler/PerfView), в Go — **встроенный `pprof`/`trace`**, позволяющий профилировать прод прямо через HTTP без сторонних агентов.
- **Отладка**: в .NET движок (`vsdbg`) интегрирован в IDE; в Go есть один отдельный движок **Delve**, над которым VS Code и GoLand — лишь GUI-фронтенды (плюс незаменимый headless-режим для контейнеров).
- **Форматирование/анализ**: `gofmt` навязывает **неконфигурируемый** канон (в отличие от `.editorconfig`-управляемого `dotnet format`), отменяя споры о стиле; анализ — `go vet` (≈ Roslyn) и `staticcheck`/`golangci-lint` (≈ StyleCop/Sonar).
- Общий вектор: **Go втягивает максимум тулинга в сам язык** (профилирование, трассировка, форматирование, базовый анализ — из коробки и одинаково у всех), вынося наружу лишь отладчик; .NET даёт богатый, но более распределённый и местами коммерческий инструментарий вокруг платформы.

На этом раздел о тулинге завершён. Вы умеете управлять зависимостями и инструментами, отлаживать через Delve и профилировать через pprof — три кита повседневной инженерной работы в Go.
