# Логирование: slog

В .NET вопрос «чем логировать» решён давно и однозначно: интерфейс `ILogger<T>`, за которым стоят провайдеры, плюс де-факто стандарт Serilog для структурного вывода. В Go путь был длиннее. Стандартный `log` из коробки умел лишь печатать строки, и экосистема годами полагалась на сторонние библиотеки. Точку поставил **Go 1.21**, добавив в стандартную библиотеку `log/slog` — структурный логгер, который стал каноном. Эта глава — про эволюцию логирования в Go и про то, как устроен и применяется `slog`.

## Эволюция: от строк к структуре

Чтобы понять, почему `slog` устроен именно так, полезно знать, через что прошла экосистема:

- **Стандартный `log`** — печатает строки в `io.Writer`. Ни уровней (кроме грубых `Print`/`Fatal`/`Panic`), ни структуры. Годится для утилит, но не для продакшн-сервиса.
- **`logrus`** (≈2014) — первый по-настоящему популярный структурный логгер. Принёс уровни и поля (`WithField`). Его слабое место — внутреннее устройство на рефлексии и аллокациях (`logrus.Fields` — это `map[string]any`), отчего он медленный и «прожорливый» под нагрузкой. Сейчас в режиме поддержки, новые проекты на нём не начинают.
- **`zap`** (Uber) и **`zerolog`** — ответ на проблему производительности. Оба заточены на **zero-allocation** логирование: типизированные поля (`zap.String("k", v)`) без боксинга в `any`, отказ от рефлексии на горячем пути. До сих пор отличный выбор там, где логирование на критическом пути и каждая аллокация на счету.
- **`log/slog`** (Go 1.21) — официальный структурный логгер в stdlib. Вобрал лучшие идеи (уровни, типизированные атрибуты, структурный вывод) и, главное, дал экосистеме **единый интерфейс** `slog.Handler`. Теперь библиотека может логировать через `slog`, а приложение — направить эти логи в любой бэкенд (в том числе в `zap`/`zerolog` через адаптер-хендлер).

> **Параллель с .NET:** эта траектория почти зеркалит .NET. Сначала был зоопарк (`log4net`, `NLog`), затем Microsoft ввела абстракцию `ILogger`/`Microsoft.Extensions.Logging` как общий контракт, а Serilog закрепил структурный подход. `slog` играет обе роли сразу: он и абстракция (`Handler` ≈ провайдер за `ILogger`), и встроенная реализация. Ключевая разница: в .NET абстракция и реализация — это пакеты, которые надо подключить и сконфигурировать через DI; в Go `slog` — часть языка «из коробки».

## Почему структурное логирование, а не форматные строки

Прежде чем код — главный сдвиг мышления. Привычное логирование — это форматные строки:

```go
// Плохо: данные «запечены» в текст
log.Printf("user %d placed order %d for $%.2f", userID, orderID, amount)
// → user 42 placed order 1007 for $19.99
```

Такую строку человек прочитает, а машина — нет. Чтобы найти все заказы пользователя `42`, придётся парсить текст регулярками, которые сломаются при первом же изменении формата. Структурное логирование отделяет **постоянное сообщение** от **переменных данных**, представляя их парами ключ-значение:

```go
// Хорошо: сообщение отдельно, данные — типизированные поля
slog.Info("order placed", "user_id", userID, "order_id", orderID, "amount", amount)
```

При выводе в JSON это превращается в машиночитаемую запись:

```json
{"time":"2026-06-20T10:15:30Z","level":"INFO","msg":"order placed","user_id":42,"order_id":1007,"amount":19.99}
```

Теперь `user_id` — это поле, по которому в Loki/Elasticsearch/Grafana строятся точные фильтры (`user_id=42`), агрегации и алерты. Это и есть смысл структурного лога: лог — не текст для глаз, а **поток событий с атрибутами** для систем сбора.

> **Параллель с .NET:** Serilog решает ту же задачу через **message templates**: `Log.Information("Order placed by {UserId} for {Amount}", userId, amount)`. Шаблон сохраняется как есть, а `{UserId}`/`{Amount}` извлекаются в структурированные свойства. Это удобнее на вид — текст и данные в одной строке. `slog` пошёл проще и явнее: никакого шаблона, ключи и значения передаются раздельно (`"user_id", userID`). Меньше «магии» парсинга шаблона — больше многословия. И там, и там итог один: структурированная запись с именованными полями.

## Базовое использование

`slog` даёт пакетные функции, работающие с логгером по умолчанию:

```go
slog.Info("server started", "port", 8080, "env", "prod")
slog.Warn("cache miss", "key", "user:42")
slog.Error("db connection failed", "err", err)
slog.Debug("payload parsed", "bytes", n) // по умолчанию не виден: уровень ниже Info
```

Сигнатура у всех одинаковая: `func(msg string, args ...any)`, где `args` — это чередующиеся ключи и значения. По умолчанию вывод идёт в текстовом формате `key=value` в stderr.

Уровни в `slog` — целые числа, что позволяет вставлять промежуточные значения:

```go
const (
    LevelDebug Level = -4
    LevelInfo  Level = 0
    LevelWarn  Level = 4
    LevelError Level = 8
)
```

> **Параллель с .NET:** `slog.Info(...)` ≈ `logger.LogInformation(...)`, `slog.Warn` ≈ `LogWarning`, и так далее. Численные уровни напоминают `LogLevel` enum, но в `slog` шаг между уровнями (4) намеренно оставляет «зазор» под кастомные уровни вроде `Notice` (между Info и Warn) — в .NET enum такого зазора не предполагает.

## `Logger` и `Handler`: ядро архитектуры

`slog` разделён на две части, и это разделение — ключ ко всему:

- **`*slog.Logger`** — фронтенд, с которым работает ваш код (`Info`, `Warn`, `With`, …). Он лишь собирает запись (`Record`) и передаёт её хендлеру.
- **`slog.Handler`** — бэкенд: решает, **куда** и **в каком формате** писать, и какие уровни пропускать. Это интерфейс:

```go
type Handler interface {
    Enabled(ctx context.Context, level Level) bool // быстрый фильтр по уровню
    Handle(ctx context.Context, r Record) error    // собственно вывод записи
    WithAttrs(attrs []Attr) Handler                // вернуть хендлер с доп. атрибутами
    WithGroup(name string) Handler                 // вернуть хендлер с группой
}
```

В стандартной библиотеке есть две готовые реализации:

- **`slog.TextHandler`** — вывод `key=value` (для локальной разработки, читается глазами).
- **`slog.JSONHandler`** — вывод JSON по строке на запись (для продакшна: машиночитаемо, дружит со сборщиками логов).

Конструируется логгер так:

```go
// JSON-логгер в stdout с минимальным уровнем Debug и указанием места вызова.
opts := &slog.HandlerOptions{
    Level:     slog.LevelDebug, // пропускать всё от Debug и выше
    AddSource: true,            // добавить file:line источника записи
}
handler := slog.NewJSONHandler(os.Stdout, opts)
logger := slog.New(handler)

logger.Info("ready", "version", "1.4.0")
```

`HandlerOptions` управляет поведением хендлера:

- **`Level`** — порог уровня (тип `slog.Leveler`; можно передать как константу, так и `*slog.LevelVar` для смены уровня в рантайме).
- **`AddSource`** — добавлять ли в запись файл и строку вызова (аналог включения source context).
- **`ReplaceAttr`** — функция-перехватчик каждого атрибута: переименовать, скрыть, отредактировать (например, замаскировать пароли или сменить формат времени).

> **Параллель с .NET:** связка `Logger` + `Handler` — это в точности `ILogger` + провайдер (`ILoggerProvider`). `Logger` — фасад, который вы вызываете; `Handler` — то, что реально пишет (как `ConsoleLoggerProvider`, файловый sink Serilog и т.п.). Принципиальная разница: в .NET к одному `ILogger` обычно подключают **несколько** провайдеров через DI, и фильтрация настраивается в `appsettings.json`. В `slog` логгер держит **один** хендлер; чтобы писать в несколько мест, используют хендлер-«размножитель» (`slog.NewMultiHandler` в свежих версиях Go или сторонний fanout-хендлер). `HandlerOptions.ReplaceAttr` ≈ Serilog `Enrich`/`Destructure` либо кастомный `ITextFormatter`.

## Типизированные атрибуты

Пары `"key", value` через `...any` удобны, но у них две слабости: значение боксится в `any` (аллокация), а опечатку «лишний/недостающий аргумент» компилятор не поймает. Для горячих путей и надёжности есть **типизированные атрибуты** — `slog.Attr`, создаваемые конструкторами:

```go
slog.Info("order placed",
    slog.Int64("user_id", userID),
    slog.String("status", "paid"),
    slog.Float64("amount", amount),
    slog.Bool("express", false),
    slog.Time("created_at", t),
    slog.Duration("elapsed", d),
    slog.Any("meta", obj), // для произвольного типа
)
```

Чтобы исключить даже малейший оверхед на интерфейсных аргументах, используют `LogAttrs` — вариант, принимающий только `[]slog.Attr` без вариативного `...any`:

```go
logger.LogAttrs(ctx, slog.LevelInfo, "order placed",
    slog.Int64("user_id", userID),
    slog.String("status", "paid"),
)
```

Связанные поля можно сгруппировать через `slog.Group` — в JSON это даст вложенный объект:

```go
slog.Info("request", slog.Group("http",
    slog.String("method", "GET"),
    slog.Int("status", 200),
))
// → ..."http":{"method":"GET","status":200}
```

> **Параллель с .NET:** `slog.String`/`Int64`/… по духу совпадают с типизированными полями `zap.String`/`zap.Int` (та же мотивация — избежать боксинга). В мире Serilog ближайшее — передача типизированных аргументов в шаблон; вложенность `slog.Group` соответствует структурированному объекту, который Serilog получает при логировании `@`-деструктурированного значения.

## `With`: контекстные поля

Часто набор полей повторяется во всех записях некоторой области — id запроса, имя компонента, user id. Вместо того чтобы дублировать их в каждом вызове, создают **производный логгер** методом `With`, который «прибивает» атрибуты ко всем последующим записям:

```go
// Базовый логгер приложения.
base := slog.New(slog.NewJSONHandler(os.Stdout, nil))

// Логгер компонента: ко всем его записям добавится component=payments.
log := base.With("component", "payments")

log.Info("charge started", "order_id", 1007)
// → ..."component":"payments","msg":"charge started","order_id":1007

// Логгер запроса: накапливаем request_id поверх компонента.
reqLog := log.With("request_id", reqID)
reqLog.Info("validated") // ..."component":"payments","request_id":...,"msg":"validated"
```

`With` возвращает **новый** `*Logger`; исходный не меняется. Под капотом он вызывает `Handler.WithAttrs`, и хороший хендлер «предформатирует» эти атрибуты один раз — поэтому `With` эффективнее, чем повторять поля в каждом вызове.

> **Параллель с .NET:** `With` функционально близок к `ILogger.BeginScope(...)` и к `Serilog` `ForContext("Component", "payments")`. Но есть тонкая разница в модели времени жизни. Scope в .NET — это `IDisposable`, привязанный к **области стека** (`using (logger.BeginScope(...))`): поля действуют, пока вы внутри `using`, и автоматически снимаются на выходе. `slog`-овский `With` привязывает поля к **экземпляру логгера** (значению), а не к области кода: поля живут ровно столько, сколько вы держите и используете этот производный логгер. Эта «привязка к значению, а не к области» — общая черта Go: вспомните, что в Разделе 3 `context.Context` тоже передавался как явное значение, а не как неявный `AsyncLocal`. `Serilog.ForContext`, возвращающий новый `ILogger`, — самый точный аналог `With`.

## `InfoContext`: проброс через `context`

В Разделе 3 мы установили, что `context.Context` пронизывает весь стек вызовов запроса. `slog` встроен в эту модель: у каждого метода есть `*Context`-вариант (`InfoContext`, `WarnContext`, `ErrorContext`, `DebugContext`), принимающий `ctx` первым аргументом:

```go
func handleOrder(ctx context.Context, id int64) error {
    slog.InfoContext(ctx, "processing order", "order_id", id)
    // ...
}
```

Зачем передавать `ctx`, если поля и так в аргументах? Потому что `ctx` доходит до `Handler.Handle(ctx, record)`, и кастомный хендлер может **извлечь данные из контекста** и подмешать их в запись — например, `trace_id` текущего спана (см. главу про OpenTelemetry) или request id, который middleware положил в контекст. Это и есть мост между логами и трейсингом: одна строка кода в хендлере связывает каждый лог с его трейсом.

Вот эскиз такого хендлера, обогащающего записи полем из контекста:

```go
type ctxHandler struct {
    slog.Handler // встраиваем базовый хендлер
}

type ctxKey struct{}

func (h ctxHandler) Handle(ctx context.Context, r slog.Record) error {
    if id, ok := ctx.Value(ctxKey{}).(string); ok {
        r.AddAttrs(slog.String("request_id", id)) // обогащаем запись
    }
    return h.Handler.Handle(ctx, r) // делегируем настоящему хендлеру
}
```

Обратите внимание: это снова паттерн **Декоратор** из первой главы раздела — оборачиваем `slog.Handler`, переопределяем `Handle`, остальное делегируем встроенному хендлеру. Тот же приём (встраивание + переопределение одного метода), что и со `statusRecorder` для `http.ResponseWriter`.

> **Параллель с .NET:** проброс `ctx` в логгер решает ту же задачу, что в .NET — `AsyncLocal<T>` и автоматическое обогащение через `Serilog.Enrichers` (например, enricher, добавляющий `TraceId` из `Activity.Current`). Разница в явности: в .NET `Activity.Current` и `AsyncLocal` — неявный ambient-контекст, текущий «сам по себе»; в Go вы **явно** передаёте `ctx` в каждый `...Context`-вызов. Больше шума в коде — меньше скрытого состояния, что в духе Go.

## Настройка логгера по умолчанию

Пакетные функции (`slog.Info` без логгера) пишут в **логгер по умолчанию**. В начале `main` его обычно подменяют на сконфигурированный, чтобы весь код (включая библиотеки, логирующие через `slog`) писал единообразно:

```go
func main() {
    logger := slog.New(slog.NewJSONHandler(os.Stdout, &slog.HandlerOptions{
        Level: slog.LevelInfo,
    }))
    slog.SetDefault(logger) // теперь slog.Info(...) везде идёт в этот JSON-хендлер

    slog.Info("application started")
    // ...
}
```

Для смены уровня в рантайме (например, временно включить Debug по сигналу, не перезапуская сервис) используют `slog.LevelVar` — потокобезопасную «ручку» уровня:

```go
levelVar := new(slog.LevelVar)        // по умолчанию LevelInfo
levelVar.Set(slog.LevelInfo)
handler := slog.NewJSONHandler(os.Stdout, &slog.HandlerOptions{Level: levelVar})
// ... позже, в обработчике admin-эндпоинта:
levelVar.Set(slog.LevelDebug)         // мгновенно поднимаем детализацию
```

> **Параллель с .NET:** `slog.SetDefault` ≈ конфигурация глобального `Log.Logger` в Serilog или построение `ILoggerFactory` на старте хоста. `LevelVar` для динамической смены уровня — аналог `LoggingLevelSwitch` в Serilog (`var levelSwitch = new LoggingLevelSwitch()` с последующим `levelSwitch.MinimumLevel = ...`).

## Логирующий middleware на `slog`

Соберём воедино с предыдущей главой: вот HTTP-middleware из паттерна Декоратор, но теперь использующий `slog` и проброс `ctx`:

```go
func LoggingMiddleware(logger *slog.Logger) Middleware {
    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            start := time.Now()
            rec := &statusRecorder{ResponseWriter: w, status: http.StatusOK}

            next.ServeHTTP(rec, r)

            logger.LogAttrs(r.Context(), slog.LevelInfo, "http request",
                slog.String("method", r.Method),
                slog.String("path", r.URL.Path),
                slog.Int("status", rec.status),
                slog.Duration("duration", time.Since(start)),
            )
        })
    }
}
```

Это та самая «луковица» из главы 1: декоратор обработчика, который теперь пишет структурный лог с типизированными атрибутами и прокидывает `r.Context()` (чтобы хендлер мог обогатить запись trace id).

## Итог

- Логирование в Go прошло путь: голый `log` → `logrus` (структурный, но рефлексивный/медленный) → `zap`/`zerolog` (zero-alloc) → встроенный **`log/slog`** (Go 1.21) как стандарт и единый интерфейс `Handler` для всей экосистемы.
- Структурное логирование = постоянное сообщение + переменные данные как пары ключ-значение. Лог — это поток событий с атрибутами для машин (фильтры, агрегации), а не текст для глаз. Аналог в .NET — message templates Serilog, но `slog` передаёт ключи и значения раздельно, без шаблона-строки.
- Архитектура — `*slog.Logger` (фронтенд) + `slog.Handler` (бэкенд: `TextHandler`/`JSONHandler`), настраиваемый через `HandlerOptions` (`Level`, `AddSource`, `ReplaceAttr`). Это `ILogger` + провайдер, но логгер держит один хендлер.
- Типизированные атрибуты (`slog.String/Int64/...`) и `LogAttrs` убирают боксинг и ловят ошибки; `slog.Group` даёт вложенность. `With` создаёт производный логгер с «прибитыми» полями — это `ForContext`/`BeginScope`, но привязка к экземпляру логгера, а не к области стека.
- `InfoContext(ctx, ...)` пробрасывает контекст до хендлера, что позволяет обогащать записи данными из `ctx` (например, `trace_id`) — мост между логами и трейсингом. Кастомный хендлер для этого — снова паттерн Декоратор.
