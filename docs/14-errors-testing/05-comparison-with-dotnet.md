# Сравнение с .NET

Эта глава консолидирует мостик всего раздела. Мы прошли обработку ошибок и тестирование Go по отдельности; теперь сведём их с .NET-стеком в одну картину — честно, с плюсами и минусами обоих подходов. Две темы здесь связаны общей философией: в .NET и обработка сбоев, и тестирование во многом опираются на рантайм и внешние фреймворки, тогда как Go выбирает явные значения и встроенный тулчейн с кодогенерацией.

## Исключения против значений-ошибок

Самое глубокое различие — модель сбоя. В .NET сбой — это **исключение**, которое возбуждается (`throw`) и автоматически «всплывает» по стеку до ближайшего `catch`, разворачивая кадры и неся стектрейс. В Go сбой — это **значение** типа `error`, которое функция возвращает, а вызывающий проверяет вручную.

```csharp
// .NET: сбой всплывает сам, точка перехвата отделена от точки выброса
try
{
    var user = repo.GetById(id);   // может бросить — но это не видно в сигнатуре
    return Format(user);
}
catch (NotFoundException ex)
{
    logger.LogWarning(ex, "не найден");
    return Default;
}
finally
{
    connection.Dispose();
}
```

```go
// Go: сбой возвращается как значение, проверка явная и локальная
user, err := repo.GetByID(ctx, id) // сигнатура честно говорит: может вернуть error
if err != nil {
    if errors.Is(err, ErrNotFound) {
        log.Printf("не найден: %v", err)
        return Default, nil
    }
    return "", err
}
defer connection.Close() // аналог finally/using — очистка при любом выходе
return Format(user), nil
```

### Плюсы и минусы — честно

**Исключения (.NET):**

- ✅ Лаконичность «счастливого пути»: основной код не перемежается проверками, ошибки уходят наверх сами.
- ✅ Богатый стектрейс из коробки — видно весь путь до места сбоя.
- ✅ Невозможно «случайно проигнорировать»: непойманное исключение уронит программу громко.
- ✅ Иерархия типов и фильтры (`catch (Ex e) when (...)`) дают выразительную селекцию.
- ❌ Поток управления **невидим**: по сигнатуре не понять, что метод бросает; выход из функции может произойти в любой строке.
- ❌ Дорого на горячем пути: разворачивание стека и аллокация объекта со стектрейсом.
- ❌ Соблазн использовать исключения для штатной логики (control flow), что считается анти-паттерном даже в .NET.

**Значения-ошибки (Go):**

- ✅ Поток управления **виден**: каждая точка сбоя и каждый «прыжок» наверх написаны явно.
- ✅ Ошибка — часть контракта (`(T, error)` в сигнатуре).
- ✅ Дёшево: нет разворачивания стека и аллокации исключения на каждый сбой.
- ✅ Ошибка — обычное значение: её легко передать, накопить (`errors.Join`), обернуть с контекстом (`%w`).
- ❌ Многословность: `if err != nil { return ..., err }` повторяется повсюду.
- ❌ Ошибку легко молча проигнорировать (`val, _ := ...`) — нужна дисциплина и линтеры (`errcheck`).
- ❌ Стектрейса в стандартной ошибке нет; контекст вы добавляете руками через оборачивание (или сторонние пакеты).

Главный вывод для .NET-разработчика: в Go **не бросают исключения для ожидаемых сбоев**. `panic` существует, но зарезервирован для невосстановимых ситуаций и багов (см. главу [Обработка ошибок](./01-error-handling.md)), а не как замена `error`.

### Таблица: исключение → как в Go

| .NET (исключения) | Go (значения-ошибки) |
| --- | --- |
| `throw new Exception("msg")` | `return errors.New("msg")` / `fmt.Errorf(...)` |
| `throw new Ex("ctx", inner)` (оборачивание) | `fmt.Errorf("ctx: %w", err)` |
| `try { ... } catch { ... }` | `if err != nil { ... }` |
| `catch (SpecificException e)` | `errors.As(err, &target)` |
| `catch` по конкретному экземпляру/маркеру | `errors.Is(err, ErrSentinel)` |
| `catch (Ex e) when (cond)` (фильтр) | `if errors.As(...) && cond { ... }` |
| `ex.InnerException` | `errors.Unwrap(err)` |
| `AggregateException` / `InnerExceptions` | `errors.Join(errs...)` (Go 1.20) |
| `finally { ... }` | `defer func() { ... }()` |
| `using var x = ...` (`IDisposable`) | `defer x.Close()` |
| `throw;` (повторный выброс) | `return err` (просто вернуть дальше) |
| свой класс-наследник `Exception` с полями | свой тип-структура с методом `Error()` |
| `ThrowIfCancellationRequested()` | `if ctx.Err() != nil { return ctx.Err() }` |
| непойманное исключение → краш | непойманная `panic` → краш |
| `catch` на верхней границе | `recover()` в `defer` на границе |

## Тест-фреймворки против встроенного `testing`

Вторая ось сравнения — тестирование. В .NET вы выбираете и подключаете фреймворк (xUnit, NUnit, MSTest), почти всегда добавляете библиотеку ассертов (FluentAssertions) и мок-библиотеку (Moq/NSubstitute). В Go раннер, базовые «ассерты» (через `t`), покрытие, бенчмарки и фаззинг **встроены** в `testing` и `go test`; внешним остаётся, по сути, лишь генератор моков и (по желанию) `testify`.

### `[Theory]` против table-driven

Параметризованные тесты — показательный пример разницы стилей. В .NET данные кейсов задают **атрибутами**; в Go — **обычным кодом** (слайс структур).

```csharp
// .NET (xUnit): данные в атрибутах
[Theory]
[InlineData(10, 2, 5)]
[InlineData(7, 1, 7)]
[InlineData(0, 5, 0)]
public void Divide_Works(int a, int b, int want)
    => Assert.Equal(want, Calc.Divide(a, b));
```

```go
// Go: данные как код, прогон в цикле с подтестами
func TestDivide(t *testing.T) {
    tests := []struct {
        name    string
        a, b    int
        want    int
    }{
        {"обычное", 10, 2, 5},
        {"на единицу", 7, 1, 7},
        {"ноль сверху", 0, 5, 0},
    }
    for _, tc := range tests {
        t.Run(tc.name, func(t *testing.T) {
            if got := Divide(tc.a, tc.b); got != tc.want {
                t.Errorf("Divide(%d,%d)=%d; want %d", tc.a, tc.b, got, tc.want)
            }
        })
    }
}
```

`[InlineData]` точнее всего ложится на table-driven для простых констант; для сложных объектов .NET использует `[MemberData]`/`[ClassData]` — а это уже ровно «данные как код», то есть тот же Go-слайс по сути. Плюс Go: кейсы можно генерировать и фильтровать программно. Плюс .NET: декларативность и интеграция с раннером IDE «из коробки».

### Moq против кодогенерации моков

Мокинг — ещё одно расхождение в механизме. Moq/NSubstitute создают подмену **в рантайме** через динамические прокси; `mockery`/`gomock` **генерируют код** из интерфейса на этапе сборки.

```csharp
// .NET: рантайм-прокси
var repo = new Mock<IUserRepository>();
repo.Setup(r => r.GetById(1)).Returns(new User("Ada"));
var svc = new UserService(repo.Object);
repo.Verify(r => r.GetById(1), Times.Once);
```

```go
// Go: сгенерированный мок (mockery), типобезопасный .EXPECT()
repo := NewMockUserRepository(t)
repo.EXPECT().GetByID(mock.Anything, 1).Return(&User{Name: "Ada"}, nil).Once()
svc := &UserService{Repo: repo}
// проверка вызовов — автоматически через t.Cleanup
```

Идея настройки похожа (`Setup().Returns()` ≈ `EXPECT().Return()`, `Verify()` ≈ автоверификация по `t`), но Go-мок — это **видимый компилируемый код**, а не рефлексивный прокси: его можно открыть, отладить, в него не «протекает» магия рантайма. Цена — генерацию надо запускать (`go generate`) и хранить результат в репозитории.

## Философия: внешний стек против встроенного

Сводя обе темы воедино, видно общий паттерн. В .NET и обработка сбоев, и тестирование во многом делегированы рантайму и экосистеме: исключения — механизм CLR, тесты — внешние фреймворки и библиотеки. В Go обе задачи решены средствами языка и тулчейна: ошибки — простые значения и пакет `errors`, тестирование — встроенный `testing`/`go test`, а недостающее (моки) добирается **кодогенерацией**, а не рантайм-рефлексией. Это та же «спартанская» линия, что проходит через весь язык: меньше магии, больше явности.

| Аспект | .NET | Go |
| --- | --- | --- |
| Модель сбоя | исключения (всплытие по стеку) | значения `error` (явный возврат) |
| Видимость сбоя в сигнатуре | нет | да (`(T, error)`) |
| Контекст ошибки | стектрейс из коробки | оборачивание `%w` вручную |
| Цепочка причин | `InnerException` | `%w` + `errors.Unwrap`/`Is`/`As` |
| Несколько ошибок | `AggregateException` | `errors.Join` (Go 1.20) |
| Очистка ресурсов | `finally` / `using` | `defer` |
| Аварийный механизм | `throw`/`catch` (общий) | `panic`/`recover` (только исключительное) |
| Тест-раннер | внешний (xUnit/NUnit/MSTest) | встроенный (`go test`) |
| Маркер теста | атрибут (`[Fact]`) | конвенция имени (`TestXxx`) |
| Параметризация | `[Theory]`/`[InlineData]`/`[MemberData]` | table-driven (слайс + `t.Run`) |
| Ассерты | внешние (FluentAssertions) или `Assert.*` | `t.Error`/`t.Fatal` или `testify` |
| Моки | рантайм-прокси (Moq/NSubstitute) | кодогенерация (`mockery`/`gomock`) |
| Бенчмарки | внешний BenchmarkDotNet | встроены (`BenchmarkXxx`) |
| Фаззинг | внешний (SharpFuzz) | встроен (`FuzzXxx`, Go 1.18) |
| Исполняемые примеры в доках | нет | `ExampleXxx` с `// Output:` |
| Интеграционные тесты | Testcontainers for .NET | `testcontainers-go` (тот же проект) |

## Итог

- Главное различие — модель сбоя: .NET бросает **исключения** (невидимое всплытие, стектрейс, дорого), Go возвращает **значения-ошибки** (явность, контракт в сигнатуре, дёшево, но многословно). В Go исключения не используют для штатных сбоев; `panic` — только для невосстановимого.
- Соответствия предсказуемы: `throw` → `return err`, `catch` → `if err != nil`, `catch (T)` → `errors.As`, `InnerException` → `%w`/`Unwrap`, `AggregateException` → `errors.Join`, `finally`/`using` → `defer`, верхний `catch` → `recover` в `defer`.
- Тест-стек: в .NET — внешние фреймворки + FluentAssertions + Moq; в Go — встроенный `testing`/`go test` с покрытием, бенчмарками, Example-тестами и фаззингом «из коробки».
- `[Theory]`/`[InlineData]` ≈ table-driven (но данные как код); Moq (рантайм-прокси) ≈ `mockery`/`gomock` (кодогенерация на этапе сборки); Testcontainers — буквально тот же проект в обеих экосистемах.
- Сквозная философия: .NET опирается на рантайм и внешние библиотеки, Go — на явные значения и встроенный тулчейн, добирая недостающее кодогенерацией вместо рантайм-магии.

На этом раздел об обработке ошибок и тестировании завершён: вы умеете возвращать и оборачивать ошибки, писать table-driven тесты, мокать через интерфейсы и поднимать реальные зависимости в Docker — и видите, чем и почему это отличается от .NET.
