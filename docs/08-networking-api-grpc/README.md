# Раздел 8. Сеть, API и Межсервисное взаимодействие

В .NET «написать сервис» почти всегда означает «взять ASP.NET Core»: Kestrel поднимет сервер, роутинг разрулят атрибуты контроллеров или Minimal API, биндинг модели, DI и фильтры приедут из коробки, а gRPC сгенерируется прямо на сборке проекта по записи `<Protobuf>` в `.csproj`. Это мощный, но непрозрачный конвейер: между вашим методом и сокетом лежит большой слой фреймворка.

Go подходит к сети с другой стороны. HTTP-сервер промышленного уровня живёт прямо в стандартной библиотеке (`net/http`) и собирается из явных кирпичиков: интерфейс `http.Handler`, маршрутизатор `ServeMux`, middleware как обычные функции. Никакого скрытого DI, никаких фильтров-атрибутов — всё, что обрабатывает запрос, вы видите в коде. gRPC же, наоборот, в Go — это **внешний** тулинг: контракт в `.proto`, отдельный шаг кодогенерации (`protoc` или, по-современному, `buf`), и явная сборка клиента/сервера из сгенерированного кода.

Этот раздел проводит вас от голого HTTP до межсервисного взаимодействия по gRPC и до того, как командой шарить контракты между сервисами. Сквозная мысль: то, что в .NET спрятано во фреймворк и SDK, в Go вынесено наружу и собирается руками — ценой многословности вы получаете полную видимость и контроль.

## Цели обучения

После этого раздела вы будете:

- Поднимать HTTP-сервер на `net/http`, понимая интерфейс `http.Handler`, `http.HandlerFunc` и `ServeMux`.
- Пользоваться улучшенным роутингом стандартной библиотеки из **Go 1.22** (паттерны с методом и wildcards: `GET /items/{id}`, `r.PathValue("id")`) и понимать, когда сторонний роутер (chi) всё ещё оправдан.
- Писать middleware как `func(http.Handler) http.Handler` и собирать их в цепочки (мостик к Разделу 11).
- Настраивать `http.Server` с таймаутами и делать graceful shutdown через `srv.Shutdown(ctx)`; грамотно переиспользовать `http.Client`.
- Описывать gRPC-контракты в `.proto`, генерировать код (`protoc-gen-go` / `protoc-gen-go-grpc`) и понимать 4 вида вызовов (unary, server/client/bidi-streaming).
- Делать client-side балансировку (`round_robin`), mTLS через `credentials` и перехватчики (interceptors).
- Шарить контракты через **Buf** (`buf generate`, `buf lint`, `buf breaking`, BSR) и держать обратную совместимость схемы.
- Осознанно выбирать между подходом ASP.NET Core «всё во фреймворке» и подходом Go «явные кирпичики».

## Ориентир: что в .NET → что в Go

Держите таблицу под рукой при чтении раздела. Детали каждого сопоставления разбираются в соответствующих главах.

| В .NET / C#                                   | В Go                                            | Ключевое отличие                                                                 |
| --------------------------------------------- | ----------------------------------------------- | ------------------------------------------------------------------------------- |
| Kestrel + хостинг ASP.NET Core                | `net/http` (`http.Server`) из stdlib            | Сервер промышленного уровня в стандартной библиотеке, без отдельного фреймворка |
| Controller / Minimal API endpoint             | `http.Handler` / `http.HandlerFunc`             | Обработчик — это просто функция `ServeHTTP(w, r)`, без базовых классов          |
| Атрибутный роутинг / `MapGet("/x/{id}")`      | `mux.HandleFunc("GET /x/{id}", ...)` (Go 1.22)  | Метод и wildcard прямо в паттерне stdlib; `r.PathValue("id")` вместо биндинга    |
| Middleware pipeline (`app.Use(...)`)          | `func(http.Handler) http.Handler` в цепочке     | Middleware — обычная функция-обёртка, порядок виден явно в коде                  |
| Фильтры (`IActionFilter`) и DI-инъекция       | Замыкания и явная передача зависимостей          | Нет контейнера: зависимости захватываются замыканием или передаются полями       |
| `IHostApplicationLifetime` + graceful shutdown | `srv.Shutdown(ctx)` по сигналу ОС               | Останов сервера — один вызов с `context` для дедлайна на дослуживание запросов   |
| `HttpClient` (+ `IHttpClientFactory`)         | `http.Client` (переиспользуемый)                | Один `Client` на приложение; таймауты задаются полем/`context`, без фабрики DI   |
| gRPC через `<Protobuf>` в `.csproj`           | `.proto` + `protoc`/`buf` (отдельный шаг)       | Кодогенерация — явный шаг, а не часть сборки проекта                             |
| Сборка генерит `*Grpc.cs` неявно              | Сгенерированные `*.pb.go` коммитятся в репо     | В Go артефакты генерации обычно лежат в git и видны в ревью                      |
| Шаринг `.proto` как NuGet-пакет               | Buf Schema Registry (BSR) / общий модуль        | Buf/BSR — язык-агностичный реестр схем вместо пакета под конкретную платформу    |
| `services.AddGrpcClient` + балансировка       | `grpc.NewClient` + `round_robin` service config | Client-side балансировка задаётся в коде через resolver и service config         |
