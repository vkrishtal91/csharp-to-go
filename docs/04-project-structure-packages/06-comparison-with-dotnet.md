# Сравнение с .NET

Эта глава собирает воедино всё, что разбиралось в разделе, и переводит его в практическую плоскость: вы держите в голове .NET-модель «решение → проекты → сборки → NuGet» и хотите быстрый ответ на вопрос «а как это устроено и как мне сделать привычное X в Go». Здесь — сводные таблицы и прямые рецепты, без повторного разжёвывания механики (она в предыдущих главах).

## Жизнь без `.sln` и `.csproj`

Главный психологический сдвиг: в Go **нет файла проекта и нет файла решения**. Нет XML, который перечисляет, какие файлы входят в компиляцию; нет действия «Add → Existing Item»; нет «выгрузить/перезагрузить проект». Структуру определяют **каталоги** и единственный `go.mod` в корне модуля.

| Понятие в .NET | Эквивалент в Go | Комментарий |
| --- | --- | --- |
| `.sln` (решение-агрегатор) | каталог репозитория + (опц.) `go.work` | Обычно агрегатора нет вовсе; границу задаёт ФС |
| `.csproj` (файл проекта) | сам факт наличия `.go`-файлов в каталоге | Нет манифеста файлов: компилируется всё в каталоге |
| «Добавить файл в проект» | просто создать `.go`-файл в каталоге | Файл подхватывается автоматически |
| «Исключить файл из сборки» | build-теги (`//go:build`) или имя `_test.go` | Управляется тегами/конвенциями имён, не манифестом |
| Сборка (assembly, `.dll`) | модуль (версии) **+** пакет (доступ) | Две роли .NET-сборки разнесены на два понятия |
| `Directory.Build.props` / общие настройки | `go.mod` (`go`-версия) + единый `gofmt` | Настроек на порядок меньше; форматирование одно для всех |

Что именно компилируется, решает сам компилятор по содержимому каталога: все `.go`-файлы каталога с одинаковым именем пакета — это пакет; файлы `*_test.go` входят только в тесты; файлы можно условно включать/исключать build-тегами (`//go:build linux`). Никакого реестра файлов вести не нужно.

### Нет merge-конфликтов в файлах проекта — схема легче

Это самое практичное следствие, которое чувствуешь каждый день в команде. В .NET `.csproj` и особенно `.sln` — это **общие файлы, которые трогает почти любое структурное изменение**: добавили класс/файл — поменялся `.csproj`; добавили проект — поменялся ещё и `.sln` (с его GUID-ами, конфигурациями сборки и порядком проектов). На активной ветвящейся разработке это постоянный источник **merge-конфликтов**: две ветки добавили по файлу — обе правили один и тот же `.csproj`/`.sln`, и git выдаёт конфликт в XML/GUID-«каше». Конфликты механические, раздражающие и легко разрешаются неправильно (потерять файл или проект при слиянии — классика).

В Go такого файла **просто нет**. Добавление `.go`-файла меняет только этот файл и не трогает никакого общего манифеста. Две ветки, добавившие разные файлы в один пакет, сливаются **без конфликта** — это разные файлы, git мержит их тривиально. Переименование и перемещение — это файловые операции, а не правки манифеста.

```text
.NET: ветка A и ветка B добавили по классу
  -> обе изменили MyApp.csproj (<Compile Include=...>) и/или .sln
  -> merge conflict ❌

Go: ветка A добавила a.go, ветка B добавила b.go в тот же пакет
  -> общих файлов не тронуто -> git мержит чисто ✅
```

И шире: вся схема **легковеснее и проще**. Нет XML-манифестов, GUID-ов, шага «добавить в проект», нет `.sln`/`.csproj`, которые надо держать в синхроне между собой и с диском. Меньше церемоний и движущихся частей, быстрее онбординг: `git clone` → `go build`, без «restore решения» и «загрузки проектов». Цена этой лёгкости — меньше явного контроля над сборкой (оговорка ниже).

> **Параллель с .NET:** отсутствие `.csproj` означает и отсутствие целого класса проблем: «файл на диске есть, но не в проекте», merge-конфликты в `.sln`/`.csproj`, рассинхрон путей. Цена — меньше явного контроля: нельзя «выключить» файл галочкой, порядок задаётся только зависимостями, а тонкие сборочные настройки задаются иначе (флаги `go build`, build-теги, переменные окружения), а не свойствами проекта.

## Модификаторы доступа

В C# четыре основных модификатора (плюс комбинации) и граница инкапсуляции — тип. В Go два уровня видимости, выражаемых регистром, и граница — пакет.

| Модификатор в C# | Ближайшее в Go | Точность соответствия |
| --- | --- | --- |
| `public` | имя с **Заглавной** буквы | Почти точно: видно всем, кто импортирует пакет |
| `private` (член класса) | — | Нет аналога: минимальная единица сокрытия — пакет, не тип |
| `internal` (в пределах сборки) | имя со **строчной** буквы (в пределах пакета) **+** каталог `internal/` | Граница — пакет/поддерево каталогов, а не сборка |
| `protected` / `private protected` | — | Нет: в Go нет наследования классов |
| `[InternalsVisibleTo("X")]` | — | Нет «белого списка»; правило `internal/` жёсткое |

Практические следствия, которые удивляют:

- Внутри одного пакета **нет приватности между типами**: метод типа `A` свободно читает строчные (приватные) поля типа `B`, если оба в одном пакете. Проектируйте границы по пакетам.
- Хотите «видно только моему модулю/поддереву» (роль `internal` из C#) — кладите код в каталог `internal/`; компилятор запретит импорт извне поддерева.
- `protected` имитировать нечем и не нужно: вместо наследования в Go — композиция (встраивание) и интерфейсы.

> **Параллель с .NET:** мысленная замена — «забудьте про `private`/`protected`, оставьте только `public` и `internal`, причём `internal`-границей считается каталог». Всё, что строчное, — это `internal`-для-пакета; всё, что под `internal/`, — это `internal`-для-модуля.

## NuGet против go-модулей

| Аспект | .NET / NuGet | Go-модули |
| --- | --- | --- |
| Источник пакетов | центральный реестр (nuget.org), приватные фиды | **напрямую из Git** (путь = адрес репо); прокси `proxy.golang.org` |
| Артефакт | собранный `.nupkg` (публикуется) | исходники в репозитории (`git tag` = публикация) |
| Манифест зависимостей | `<PackageReference>` в `.csproj` | `require` в `go.mod` |
| Фиксация целостности | `packages.lock.json` (`contentHash`) | `go.sum` (хеши) + checksum database |
| Хранение выбранных версий | в lock-файле | **нигде**: вычисляется детерминированно (MVS) |
| Кеш | `~/.nuget/packages` + restore в проект | один глобальный `~/go/pkg/mod` (read-only) |
| Алгоритм версий | разрешение диапазонов (max, совместимый со всеми) | **MVS**: наибольшая *затребованная* версия |
| Мажорные версии рядом | конфликт (binding redirects/ALC) | сосуществуют (`/v2` в пути импорта) |
| Команда «добавить» | `dotnet add package X` | `go get X@version` |
| Команда «привести в порядок» | `dotnet restore` | `go mod tidy` |
| Офлайн-копия в репозитории | редко (`packages/` в прошлом) | штатный `go mod vendor` → `vendor/` |

Два различия стоит подчеркнуть как самые «контринтуитивные»:

1. **Нет реестра — есть Git.** Вы не публикуете артефакт; вы пушите тег. Любой Git-репозиторий с `go.mod` — уже импортируемая зависимость. Приватные зависимости настраиваются через `GOPRIVATE` (аналог приватного фида), а не через публикацию в закрытый реестр.
2. **Версии не «плавают».** MVS детерминирован и консервативен: вышедшая новая версия зависимости **не** попадёт в вашу сборку, пока вы явно не запросите `go get -u`. Это противоположно привычке к плавающим версиям/диапазонам и убирает «у меня собралось, у тебя нет».

> **Параллель с .NET:** если в .NET вы привыкли мыслить «найди мне максимально свежее в рамках диапазона», то в Go мышление — «у меня зафиксирован точный минимум, обновляю осознанно». Lock-файл в Go не нужен для повторяемости, потому что повторяемость даёт сам алгоритм; `go.sum` отвечает лишь за то, чтобы скачанный код не подменили.

## Project references против импортов пакетов

В .NET связи между своими проектами — это `<ProjectReference>` (граф между сборками, который тоже обязан быть ациклическим). В Go никакого отдельного объявления связей нет: вы просто **импортируете пакет по пути**, и связь возникает.

| .NET | Go |
| --- | --- |
| `<ProjectReference Include="..\Lib\Lib.csproj" />` | `import "github.com/acme/app/internal/lib"` |
| Граф сборок (ацикличный между сборками) | Граф пакетов (**ацикличный между пакетами** — строже) |
| Циклы внутри сборки разрешены (классы) | Циклы внутри пакета разрешены (типы), между пакетами — **нет** |
| Разрыв цикла сборок: выделить общую сборку | Разрыв цикла пакетов: общий пакет **или** интерфейс у потребителя |

Ключевое отличие в строгости: в .NET два класса `Order` и `User` в **одной** сборке свободно ссылаются друг на друга. В Go, если они в **разных** пакетах, взаимный импорт — ошибка компиляции; нужно либо положить их в один пакет, либо инвертировать зависимость через неявно реализуемый интерфейс (см. главу про модули). Это дисциплинирует архитектуру сильнее, чем .NET.

## Рецепты: «как мне сделать привычное X»

| Привычная задача из .NET | Решение в Go |
| --- | --- |
| Новый «проект» под программу | каталог с `main.go` (`package main`); для нескольких бинарей — `cmd/<app>/main.go` |
| Class Library (переиспользуемая) | **пакет** (каталог), если внутри того же модуля; **отдельный модуль** (`go.mod`), если версионируется/публикуется независимо |
| Несколько проектов в одном решении | один модуль с несколькими `cmd/<app>/` + общий `internal/`; либо несколько модулей, объединённых `go.work` локально |
| Скрыть код в пределах сборки (`internal`) | каталог `internal/` (компилятор запретит импорт извне поддерева) |
| Скрыть член класса (`private`) | строчная первая буква (видно в пределах пакета — это минимальная единица сокрытия) |
| Публичный API библиотеки | имена с Заглавной буквы; внутренности — в `internal/` или строчные |
| Подключить NuGet-пакет | `go get <import-path>@<версия>` |
| Локальная отладка зависимости | `replace ... => ../local` в `go.mod` **или** `go.work` с `use` |
| Восстановить пакеты в CI | `go mod download` (прогрев кеша) |
| Lock-файл для повторяемых сборок | уже есть: `go.mod` (версии) + `go.sum` (хеши); ничего дополнительно делать не нужно |
| Офлайн/герметичная сборка | `go mod vendor` → каталог `vendor/` |
| Перейти на мажорную версию пакета | сменить путь импорта на `/v2`, `/v3` (а не только номер версии) |
| Запустить статический инициализатор сборки | `init()` в пакете; для регистрации через сторонний пакет — пустой импорт `_ "pkg"` |

Главный вывод для миграции: перестаньте проектировать «по проектам/сборкам» и начните «по пакетам/каталогам». Большинство случаев, где в .NET вы завели бы новый `.csproj`, в Go решаются новым **каталогом-пакетом** внутри одного модуля. Новый **модуль** заводят реже — только когда нужно независимое версионирование и распространение (то есть когда вы реально публикуете отдельно версионируемую библиотеку).

## Итог

- **Нет `.sln`/`.csproj`**: структуру задают каталоги и один `go.mod`; компилируется всё в каталоге, без манифеста файлов и без «добавить в проект». Как следствие — **нет merge-конфликтов в файлах проекта при слиянии веток**, и вся схема легче и проще.
- **Доступ через регистр, граница — пакет**: Заглавная ≈ `public`, строчная ≈ «в пределах пакета»; роль `internal` играет каталог `internal/`; `private`/`protected` аналогов не имеют.
- **Модули вместо NuGet**: код из Git без обязательного реестра, целостность через `go.sum`, версии детерминированно через MVS (не плавают), мажоры сосуществуют через `/v2` в пути.
- **Импорт вместо project reference**: связь возникает из `import`; граф пакетов обязан быть ацикличным — строже, чем граф сборок в .NET.
- **Сдвиг проектирования**: «новый проект» → чаще «новый каталог-пакет в том же модуле»; отдельный модуль — только для независимо версионируемой/публикуемой библиотеки.
