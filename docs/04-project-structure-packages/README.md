# Раздел 4. Структура проекта, Пакеты и Зависимости

В .NET физическая организация кода жёстко завязана на файлы проекта и решения: `.sln` собирает воедино набор `.csproj`, каждый `.csproj` компилируется в отдельную сборку (assembly), а сборка — это и единица развёртывания, и граница доступа (`internal`), и узел графа зависимостей. Вы привыкли мыслить «проектами»: создать `ClassLibrary`, добавить project reference, расставить модификаторы доступа, подтянуть NuGet-пакет, восстановить зависимости перед сборкой.

В Go этого слоя просто нет. Нет файла проекта, нет «добавить файл в проект», нет понятия «сборка» как явного артефакта компиляции с границей доступа. Структуру задают **каталоги файловой системы** и один файл `go.mod` на весь модуль. Каталог — это пакет (единица компиляции и инкапсуляции), регистр первой буквы имени — это модификатор доступа, а зависимости тянутся напрямую из Git и кешируются локально, без центрального реестра в обязательном порядке. Этот раздел перенастраивает вашу «проектную» интуицию на «пакетно-каталожную» модель Go и разбирает три кита: как раскладывать код по каталогам, как работает видимость на уровне пакетов и как устроено управление зависимостями через go-модули.

## Цели обучения

После этого раздела вы будете:

- Понимать, что `golang-standards/project-layout` — это **не** официальный стандарт, и раскладывать код так, как реально рекомендует команда Go: от плоской структуры к `cmd/`/`internal/` по мере роста.
- Знать, что `internal/` — это **механизм языка**, а не конвенция: импорт такого кода извне поддерева запрещается компилятором.
- Понимать модель видимости Go (экспорт через заглавную букву, пакет как единицу инкапсуляции) и сопоставлять её с `public`/`private`/`internal`/`protected` из C#.
- Уверенно работать с `go.mod`/`go.sum`, командами `go get`/`go mod tidy`/`go build`, понимать Semantic Import Versioning (`/v2` в пути) и алгоритм MVS в отличие от резолвера NuGet.
- Знать, что циклические импорты между пакетами — это **ошибка компиляции**, и уметь их разрывать через инверсию зависимостей.
- Переводить привычные .NET-решения («несколько проектов в решении», «class library», «project reference») на идиомы Go.

## Ориентир: что в .NET → что в Go

Держите таблицу под рукой при чтении раздела. Детали каждого сопоставления разбираются в соответствующих главах.

| В .NET / C# | В Go | Ключевое отличие |
| --- | --- | --- |
| `.sln` (решение) | каталог репозитория + `go.mod` | Нет файла-агрегатора; границы задаёт файловая система |
| `.csproj` (проект → сборка) | пакет (каталог) внутри модуля | Пакет — не артефакт развёртывания; единица компиляции и инкапсуляции |
| Сборка (assembly, `.dll`) | модуль (граница версионирования) / пакет (граница доступа) | В Go две разные роли разнесены: версионирует модуль, инкапсулирует пакет |
| `public` / `private` / `internal` | заглавная буква (экспорт) / строчная (приватно) + папка `internal/` | Видимость — свойство имени и расположения, а не ключевое слово |
| Project reference (`<ProjectReference>`) | `import "module/path/pkg"` | Импортируется пакет по пути, не «проект» |
| NuGet (центральный реестр, `.nupkg`) | `go get` напрямую из Git | Нет обязательного реестра; источник — VCS-репозиторий |
| `packages.lock.json` / restore | `go.sum` + модульный кеш | Lock — это контрольные суммы; «восстановление» = докачка в кеш |
| Циклы между сборками запрещены, внутри сборки — разрешены | Циклы между пакетами запрещены (ошибка компиляции) | Запрет в Go строже: действует на уровне пакета, а не сборки |
| Несколько проектов в одном решении | `cmd/<app>/` (много бинарей) или несколько модулей | Один модуль легко даёт много исполняемых файлов |
