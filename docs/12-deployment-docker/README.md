# Раздел 12. Развертывание, Docker и Инфраструктура

В мире .NET «развернуть сервис» долгое время означало «привезти на машину рантайм нужной версии и положить рядом сборки». Контейнеризация это упростила, но образ всё равно носит с собой .NET runtime: даже минимальный ASP.NET-сервис тащит за собой managed-рантайм, JIT и базовые библиотеки. Go меняет саму единицу развёртывания. Компилятор Go выдаёт **один статически слинкованный бинарник** без внешних зависимостей — его можно положить в пустой образ `scratch` и запустить там, где нет ни рантайма, ни libc, ни shell.

Этот сдвиг меняет и приоритеты: в Go-сообществе «маленький образ и мгновенный старт» — предмет инженерной гордости, а multi-stage-сборка и выбор базового образа (`scratch` / `distroless` / `alpine`) — обязательная часть культуры, а не экзотика. Этот раздел перенастраивает вашу «развёрточную» интуицию с модели «рантайм + артефакты» на модель «один самодостаточный бинарник» и разбирает, как из исходников Go получить компактный, безопасный и быстро стартующий контейнер.

Раздел практический: основной вес — на устройстве эффективного Docker-образа (multi-stage, статическая линковка, кэширование слоёв), затем игнор-файлы (`.gitignore`/`.dockerignore`), и в конце — консолидированное сравнение с тем, как это устроено в .NET.

## Цели обучения

После этого раздела вы будете:

- Собирать **multi-stage** Docker-образ: стадия сборки на `golang:1.x` → финальная стадия с единственным бинарником, без компилятора и исходников в итоговом образе.
- Понимать статическую линковку Go: зачем `CGO_ENABLED=0`, что дают `-ldflags="-s -w"`, и как делать кросс-компиляцию через `GOOS`/`GOARCH`.
- Осознанно выбирать базовый образ — `scratch`, `distroless` или `alpine` — и знать, что для каждого надо доложить вручную (`ca-certificates`, `tzdata`, часовые пояса).
- Правильно кэшировать слои: сначала `go.mod`/`go.sum` + `go mod download`, потом исходники; использовать `--mount=type=cache` для кэша компилятора.
- Писать корректные `.dockerignore` и `.gitignore` для Go-проекта и понимать, что и почему из них исключать.
- Объяснять, почему Go-образ на порядок меньше ASP.NET-образа и почему AOT-бинарник Go стартует мгновенно в отличие от JIT-warmup у .NET (и где Native AOT сближает .NET с этой моделью).

## Ориентир: что в .NET → что в Go

Держите таблицу под рукой при чтении раздела. Детали каждого сопоставления разбираются в соответствующих главах.

| В .NET | В Go | Ключевое отличие |
| --- | --- | --- |
| `dotnet publish` → набор `.dll` + рантайм | `go build` → один бинарник | В Go артефакт — единственный self-contained файл |
| Базовый образ `mcr.microsoft.com/dotnet/aspnet` | `scratch` / `distroless` / `alpine` | Go-бинарнику базовый образ часто не нужен вовсе |
| .NET runtime внутри образа (JIT) | ничего, кроме бинарника (AOT) | Нет managed-рантайма; старт мгновенный |
| Multi-stage: SDK-образ → runtime-образ | Multi-stage: `golang` → `scratch`/`distroless` | Идея одна; финальная стадия в Go радикально тоньше |
| Self-contained / Native AOT publish | поведение Go по умолчанию | То, что в .NET опция, в Go — норма |
| `.dockerignore` исключает `bin/`, `obj/` | `.dockerignore` исключает бинарники, `.git`, кэш | Те же мотивы: скорость, размер, безопасность |
| `.gitignore`: `bin/`, `obj/` | `.gitignore`: бинарники, `*.out`, `.env`; нюанс с `vendor/` | Артефакты сборки не коммитят в обоих мирах |
| NuGet-кэш (`~/.nuget/packages`) | модульный кэш (`$GOPATH/pkg/mod`) + `--mount=type=cache` | В обоих образ не должен тащить локальный кэш |
