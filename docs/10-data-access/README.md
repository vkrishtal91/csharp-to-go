# Раздел 10. Работа с Данными

В .NET доступ к данным — это, как правило, история про **тяжёлые абстракции**: Entity Framework Core с change tracking, миграциями и LINQ-провайдером, который переводит выражения в SQL за вас. Даже когда вы спускаетесь ниже, к Dapper, вы остаётесь в мире «дай мне строку SQL и тип — я сам разложу результат по объектам». Под всем этим лежит ADO.NET (`DbConnection`/`DbCommand`/`DataReader`) с прозрачным, почти невидимым пулом соединений, которым управляет провайдер.

В Go культура доступа к данным принципиально другая, и это первое, что нужно перенастроить в голове. Стандартная библиотека даёт `database/sql` — это **не ORM и даже не клиент конкретной СУБД**, а тонкая абстракция над драйверами с одной по-настоящему важной вещью внутри: пулом соединений. Поверх неё сообщество строит лёгкие надстройки (`sqlx`, `sqlc`), которые убирают рутину ручного маппинга, но **не прячут SQL**. Полноценный ORM (`gorm`) существует, но в Go-сообществе он — осознанный выбор меньшинства, а не путь по умолчанию. Доминирующая философия: оставаться близко к SQL, писать запросы явно, платить за абстракцию сознательно.

Этот раздел перенастраивает вашу «EF-интуицию» на модель Go: сначала разбираем `database/sql` и его пул (ключевая глава), затем `sqlx` как «Dapper по духу», потом Redis и паттерн Cache-Aside, и в конце — консолидированное сравнение с EF Core и Dapper.

## Цели обучения

После этого раздела вы будете:

- Понимать, что `database/sql` — это **абстракция над драйверами**, а не ORM, и что `sql.DB` — это **пул соединений** (а не одно соединение), потокобезопасный и создаваемый один раз на всё приложение.
- Уметь настраивать пул (`SetMaxOpenConns`, `SetMaxIdleConns`, `SetConnMaxLifetime`, `SetConnMaxIdleTime`) и понимать, чем это отличается от неявного pooling в ADO.NET.
- Правильно выполнять запросы через `QueryContext`/`QueryRowContext`/`ExecContext` с обязательным `context`, корректно закрывать `rows`, проверять `rows.Err()` и обрабатывать `NULL`.
- Вести транзакции (`BeginTx`/Commit/Rollback) с идиомой `defer rollback`.
- Использовать `sqlx` для устранения рутины ручного `Scan` (`Get`/`Select`/`StructScan`/`NamedExec`) и понимать, где проходит граница между `sqlx`, `sqlc` и `gorm`.
- Применять `go-redis/v9` и реализовывать паттерн Cache-Aside (lazy loading) с TTL, осознавая подводные камни (инвалидация, cache stampede).
- Переводить привычные .NET-практики (`DbContext`, LINQ-запрос, миграции) на идиомы Go.

## Ориентир: что в .NET → что в Go

Держите таблицу под рукой при чтении раздела. Детали каждого сопоставления разбираются в соответствующих главах.

| В .NET / C# | В Go | Ключевое отличие |
| --- | --- | --- |
| ADO.NET (`DbConnection`/`DbCommand`/`DataReader`) | `database/sql` (`*sql.DB`/`*sql.Rows`) | Та же роль «низкоуровневый доступ», но в Go это часть стандартной библиотеки |
| Пул соединений ADO.NET (неявный, в строке подключения) | `*sql.DB` (явная настройка лимитов) | Пул в Go настраивается кодом, а не параметрами connection string |
| Провайдер (`Npgsql`, `Microsoft.Data.SqlClient`) | драйвер (`pgx`, `lib/pq`, `go-sql-driver/mysql`) | Драйвер импортируется ради side-effect (`_ "..."`) |
| `DbDataReader.Read()` + индексаторы | `rows.Next()` + `rows.Scan(&...)` | В Go маппинг строки в поля — вручную (или через `sqlx`) |
| `DbCommand.Parameters` (`@p`) | плейсхолдеры драйвера (`$1` для Postgres, `?` для MySQL) | Синтаксис плейсхолдера зависит от драйвера, не от языка |
| `DbTransaction` (`BeginTransaction`) | `*sql.Tx` (`BeginTx`) | Идиома `defer tx.Rollback()` вместо `using` |
| Dapper (`QueryAsync<T>`, `QueryFirstOrDefault<T>`) | `sqlx` (`Select`, `Get`) | Микро-ORM «маппинг строк в структуры», очень близко по духу |
| EF Core (`DbContext`, LINQ, change tracking) | `gorm` (или `sqlc` как анти-ORM) | В Go полноценный ORM — выбор меньшинства; чаще остаются близко к SQL |
| Миграции EF (`Add-Migration`/`Update-Database`) | `goose` / `golang-migrate` | Отдельные инструменты, не часть библиотеки доступа |
| `StackExchange.Redis` / `IDistributedCache` | `redis/go-redis/v9` | Тот же набор операций; Cache-Aside реализуется одинаково |
