# Zed для Go

Zed — редактор, написанный на Rust, с упором на скорость (мгновенный старт, отзывчивый ввод даже на больших файлах, GPU-ускоренный рендеринг) и совместную работу. Для Go он привлекателен ровно тем же, чем и для других языков: он быстрый и при этом использует тот же стандартный тулчейн, что и все остальные редакторы. Языковой интеллект в Zed обеспечивает всё тот же `gopls`, поэтому навыки и настройки `gopls` переносятся сюда без изменений — меняется лишь формат конфигурации и способ описания отладки.

Эта глава короче главы про VS Code, но самодостаточна: настройка `gopls`, текущее состояние отладки через Delve и честное сравнение с VS Code для серверной работы. Базовые инструменты считаем установленными по [обзору раздела](./README.md).

## Поддержка Go: `gopls` под капотом

У Zed встроенный LSP-клиент, и он сам знает, что для `.go`-файлов нужно запустить `gopls` — большинство возможностей (автодополнение, переходы, диагностика, переименования) работает «из коробки», без расширений.

Важный нюанс, в отличие от VS Code: **Zed не ставит `gopls` автоматически**. Официальная документация прямо рекомендует установить его через `go install` (а не через Homebrew или пакетный менеджер дистрибутива) и убедиться, что бинарник в `PATH`:

```bash
go install golang.org/x/tools/gopls@latest
# Убедитесь, что ~/go/bin в PATH (см. обзор раздела), иначе Zed не найдёт gopls:
export PATH="$PATH:$(go env GOPATH)/bin"
```

> **Параллель с .NET:** как и в случае VS Code, опыт работы с языковым интеллектом переносим, потому что «мозг» один и тот же — `gopls` (по духу аналог Roslyn/OmniSharp). Zed — редактор не про .NET-стек, но именно из-за общего язык-сервера ваши Go-настройки не привязаны к редактору: то, что вы настроили для VS Code через `gopls`, концептуально один-в-один кладётся в конфиг Zed.

## Настройки Zed для Go

Конфигурация Zed — это тоже `settings.json` (открывается командой `zed: open settings`), но его структура отличается от VS Code: настройки языка живут под ключом `languages`, а параметры язык-сервера передаются в `lsp.<server>.initialization_options`. Для `gopls` это означает, что знакомые опции (`gofumpt`, `staticcheck`, inlay hints) кладутся именно в `initialization_options`, повторяя структуру настроек самого `gopls`.

Готовый блок с пояснениями:

```jsonc
{
  // Форматировать при сохранении. Для Go форматтером выступает gopls
  // (через LSP), поэтому отдельный formatter указывать не нужно.
  "format_on_save": "on",

  // Настройки, специфичные для языка Go.
  "languages": {
    "Go": {
      // Организация импортов при сохранении: gopls удалит лишние и
      // добавит недостающие импорты (аналог organizeImports в VS Code).
      "code_actions_on_format": {
        "source.organizeImports": true
      }
    }
  },

  // Конфигурация языковых серверов.
  "lsp": {
    "gopls": {
      // Параметры передаются gopls при инициализации. Структура повторяет
      // настройки самого gopls (те же имена, что в его документации).
      "initialization_options": {
        // Строгий форматтер gofumpt прямо в gopls (бинарник не требуется).
        "gofumpt": true,

        // Включить диагностику staticcheck по мере набора.
        "staticcheck": true,

        // Точечно включить/выключить анализаторы gopls.
        "analyses": {
          "unusedparams": true,   // неиспользуемые параметры функций
          "shadow": true          // затенение переменных (var shadowing)
        },

        // Inlay hints — встроенные подсказки в коде (имена параметров,
        // выведенные типы и т.п.). По умолчанию выключены — включаем нужные.
        "hints": {
          "assignVariableTypes": true,
          "compositeLiteralFields": true,
          "compositeLiteralTypes": true,
          "constantValues": true,
          "functionTypeParameters": true,
          "parameterNames": true,
          "rangeVariableTypes": true
        }
      }
    }
  },

  // Чтобы inlay hints от gopls реально отображались, в Zed их нужно
  // также включить на уровне редактора:
  "inlay_hints": {
    "enabled": true
  }
}
```

Два момента, где Zed отличается от VS Code по форме:

- **Имена опций `gopls` — «родные».** В VS Code расширение оборачивает их в свои ключи (`gopls.ui.inlayhint.hints`, `gopls.formatting.gofumpt`). В Zed вы пишете опции `gopls` напрямую в `initialization_options` (`gofumpt`, `staticcheck`, `hints`, `analyses`) — это буквально то, что понимает сам `gopls`. Поэтому при переносе настроек между редакторами держите в голове: меняется обёртка, а не сами имена параметров язык-сервера.
- **Inlay hints включаются в двух местах.** Сами категории подсказок задаются в `gopls` (`hints`), но их показ ещё управляется опцией редактора `inlay_hints.enabled` — без неё gopls будет их вычислять, а Zed не покажет.

> **Линтер.** Полноценная интеграция `golangci-lint` как отдельного диагностического источника в Zed менее «из коробки», чем в VS Code (где есть готовый `go.lintTool`). На практике многие полагаются на диагностику `gopls` + `staticcheck` (включается опцией выше), а полный `golangci-lint` гоняют через задачу/таск или pre-commit/CI. Если нужна именно инлайн-интеграция линтера — сверьтесь с актуальной документацией Zed, тут детали со временем меняются.

## Отладка в Zed через Delve

Здесь — самая «свежая» и быстро меняющаяся часть, поэтому отметим честно: **встроенный отладчик Zed появился недавно** — он был представлен 18 июня 2025 года и подан командой как «большой шаг к Zed 1.0». Это уже не эксперимент, а штатная функция: точки останова, инспекция переменных, шаги, управление сессией. Но он заметно моложе отладки в VS Code/GoLand, и команда открыто продолжает его развивать (обещаны watch-list, просмотр памяти, дизассемблер, расширение списка языков). Поэтому ожидайте, что отдельные детали будут меняться — сверяйтесь с актуальной документацией.

Как и везде, Zed отлаживает Go через **Delve** по протоколу DAP — в конфигурации это адаптер `"Delve"`. Поддерживается отладка «из коробки» без конфигурации: команда **`debugger: start`** (по умолчанию `F4`) показывает контекстный список преднастроенных сценариев — она сама находит точки входа (`func main`) и тесты и предлагает их запустить. Для большинства локальных случаев этого достаточно.

Для повторяемых сценариев конфигурации описываются в файле `.zed/debug.json` — массиве объектов. Поля по смыслу близки к `launch.json` VS Code, но с другими именами и оформлением.

Запуск точки входа сервера:

```jsonc
[
  {
    "label": "Debug API server",
    "adapter": "Delve",
    "request": "launch",
    "mode": "debug",
    // В режиме debug program указывает на пакет с func main.
    "program": "./cmd/api",
    "cwd": "$ZED_WORKTREE_ROOT",
    "args": ["--addr", ":8080"],
    // Build-флаги/теги передаются отдельным полем (аналог buildFlags в VS Code).
    "buildFlags": ["-tags", "integration"],
    // Переменные окружения — для конфигурации сервера (12-factor).
    "env": {
      "APP_ENV": "local",
      "LOG_LEVEL": "debug"
    }
  }
]
```

Отладка тестов пакета (с build-тегом):

```jsonc
[
  {
    "label": "Debug integration tests",
    "adapter": "Delve",
    "request": "launch",
    "mode": "test",
    "program": "./internal/order",
    "buildFlags": ["-tags", "integration"]
  }
]
```

Подключение к уже запущенному headless-серверу Delve (в т.ч. в контейнере) — через `tcp_connection`:

```jsonc
[
  {
    "label": "Attach to dlv in Docker",
    "adapter": "Delve",
    "request": "launch",
    "mode": "exec",
    // Подключение к headless-dlv: порт проброшен из контейнера на localhost.
    "tcp_connection": { "host": "127.0.0.1", "port": 40000 }
  }
]
```

Сервер Delve внутри контейнера поднимается ровно той же командой, что и для VS Code (см. [главу про VS Code](./01-vscode.md) и [Раздел 12](../12-deployment-docker/README.md)):

```bash
dlv debug ./cmd/api --headless --listen=:40000 --api-version=2 --accept-multiclient
```

> Та же оговорка по безопасности, что и везде: открытый порт Delve = полный контроль над процессом. Только приватная сеть/туннель, только на время отладки.

Поскольку отладчик Zed молод, набор полей и нюансы удалённой/attach-отладки для Go стоит сверять с [официальной документацией Zed](https://zed.dev/docs/debugger) — это та область, где формулировки и возможности активно дорабатываются.

## Zed против VS Code для серверной Go-разработки

Оба редактора используют один и тот же `gopls` и один и тот же Delve, поэтому базовый опыт письма на Go близок. Разница — в зрелости периферии и в скорости.

| Критерий | Zed | VS Code (`golang.go`) |
| --- | --- | --- |
| Язык-сервер | `gopls` (ставится вручную) | `gopls` (ставится через «Install/Update Tools») |
| Скорость/отзывчивость | ✅ очень быстрый (Rust, GPU) | медленнее на больших проектах |
| Отладка (Delve/DAP) | есть, но молодая (с июня 2025), активно дорабатывается | ✅ зрелая, все режимы, годы шлифовки |
| Интеграция линтера | базово через `gopls`/`staticcheck`; `golangci-lint` — обычно через таск/CI | ✅ готовый `go.lintTool` для `golangci-lint` |
| Экосистема/настройки | моложе, меньше готовых рецептов | ✅ обширная, много документации и примеров |
| Изоляция окружений | проекты/настройки | ✅ профили (изоляция расширений от C#/JS) |

Кому что подойдёт:

- **Zed** — если вы цените скорость и минимализм, пишете в основном Go (и, может, ещё пару LSP-языков), а сложная отладка вам нужна редко или вы спокойно спускаетесь к `dlv` из терминала. Чем дальше, тем сильнее закрывается разрыв в отладке.
- **VS Code** — если вам важны зрелая, «кликабельная» отладка всех мастей (особенно удалённая в контейнерах), готовая интеграция линтера и большой запас готовых рецептов и расширений; а также если вы держите на одной машине несколько стеков и хотите изолировать их профилями. Для пришедшего из Visual Studio это и наиболее знакомый по ощущениям путь.

Разумная стратегия — не выбирать раз и навсегда: повседневное редактирование вести там, где приятнее (нередко это Zed за счёт скорости), а под тяжёлую отладочную сессию открыть проект в VS Code. Тулчейн общий, поэтому переключение почти бесплатно.

## Итог

- Zed — быстрый редактор на Rust; языковой интеллект для Go даёт тот же `gopls`, поэтому настройки и навыки `gopls` переносятся из любого редактора.
- В отличие от VS Code, Zed **не ставит `gopls` сам** — нужен `go install golang.org/x/tools/gopls@latest` и `PATH`.
- Настройки — в `settings.json` Zed: `format_on_save`, `code_actions_on_format` → `source.organizeImports`, а параметры `gopls` (`gofumpt`, `staticcheck`, `analyses`, `hints`) кладутся в `lsp.gopls.initialization_options` под их «родными» именами; inlay hints дополнительно включаются через `inlay_hints.enabled`.
- Отладка через **Delve** (адаптер `"Delve"`): zero-config через `debugger: start` (F4) и конфиги в `.zed/debug.json` (`request`/`mode`/`program`/`cwd`/`args`/`buildFlags`/`env`, attach к headless-`dlv` через `tcp_connection`). Отладчик появился в июне 2025 — рабочий, но моложе VS Code и активно развивается; детали стоит сверять с документацией.
- Выбор: **Zed** — за скорость и минимализм при умеренных требованиях к отладке; **VS Code** — за зрелую отладку (включая удалённую в Docker), готовую интеграцию линтера, профили и обширную экосистему. Общий тулчейн делает переключение между ними почти бесплатным.
