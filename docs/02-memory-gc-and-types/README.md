# Раздел 2. Память, GC и Система типов (Глубокое погружение)

В .NET вопрос «где живёт мой объект» решается выбором между `struct` и `class`: первый по умолчанию на стеке или инлайном, второй — всегда в управляемой куче. Этот выбор вы делаете руками, явно, на этапе объявления типа. В Go такого дуализма нет: вы объявляете один `struct`, а решение о размещении в стеке или куче принимает компилятор сам, на основе анализа того, «убегает» ли значение за пределы своего фрейма (escape analysis). Это первый и самый важный сдвиг мышления в этом разделе.

Дальше мы пройдём по всей системе типов Go через призму вашего .NET-опыта: указатели и передача по значению (всё копируется, всегда), нулевые значения вместо `null` и почему `NullReferenceException` превращается в `nil panic`, неявная (структурная) реализация интерфейсов вместо явного `: IInterface`, внутреннее устройство интерфейса как пары (тип, значение), боксинг при упаковке в `any` и как дженерики (Go 1.18+) его убирают. Венчает раздел детальный разбор сборщика мусора Go — конкурентного, непоколенческого, неперемещающего — и консолидированное сравнение с поколенческим уплотняющим GC из .NET.

## Цели обучения

После этого раздела вы будете:

- Понимать, что такое escape analysis, и читать вывод `go build -gcflags='-m'`, чтобы видеть, что компилятор разместил в куче, а что оставил на стеке.
- Осознанно выбирать между передачей по значению и по указателю, не впадая в карго-культ «всё по указателю».
- Знать таблицу нулевых значений наизусть, понимать идиому «полезного нулевого значения» и не попадаться на классическую ловушку «typed nil в интерфейсе».
- Объяснять, почему методы с pointer receiver можно вызывать на `nil`-указателе, и понимать внутреннее устройство интерфейса (eface/iface, itab).
- Понимать боксинг при упаковке в `any`, безопасно извлекать значения через type assertion / type switch, и писать дженерик-код с constraints, `comparable` и оператором `~`.
- Точно описывать алгоритм сборщика мусора Go (трёхцветный конкурентный mark-and-sweep с write barrier) и грамотно сравнивать его с поколенческим компактящим GC из .NET.

## Ориентир: что в .NET → что в Go

| В .NET / C# | В Go | Ключевое отличие |
|---|---|---|
| `struct` (стек/инлайн) vs `class` (всегда куча) — выбор вручную | Один `struct`; размещение решает escape analysis | Контроль над стеком/кучей в Go неявный, его делает компилятор, а не разработчик |
| Generational compacting GC (gen0/1/2, LOH) | Конкурентный non-moving, non-generational mark-and-sweep | Go не уплотняет и не перемещает объекты; оптимизирован под короткие паузы |
| `NullReferenceException` (NRE) | `nil pointer dereference` → паника + нулевые значения | Многих NRE нет: неинициализированные ссылочные типы получают полезный zero value |
| Явная реализация интерфейса (`class C : IFoo`) | Неявная (структурная) реализация | Тип реализует интерфейс автоматически по набору методов; `: IInterface` не нужен |
| `object` + боксинг value-типов | `any` (= `interface{}`) + боксинг | Упаковка значения в интерфейс обычно вызывает аллокацию; дженерики её убирают |
| Nullable reference types (C# 8, защита на компиляции) | Нулевые значения + линтеры | В Go нет защиты от nil на уровне типов; спасают zero values, дисциплина и статанализ |
