# Сравнение с .NET

Эта глава собирает воедино то, что в предыдущих разбиралось по частям, и наводит прямой мост между сериализацией в .NET и в Go. Главный фокус — JSON, потому что именно там накоплен ваш самый плотный .NET-опыт и больше всего различий в моделях. Если предыдущие главы отвечали на вопрос «как это устроено в Go», то эта отвечает на вопрос «я умею делать X в .NET — как сделать то же в Go».

## Два мира .NET: `System.Text.Json` и `Newtonsoft.Json`

В .NET у вас исторически два главных JSON-сериализатора, и важно понимать, на что в Go похож каждый:

- **`System.Text.Json` (STJ)** — современный дефолт (с .NET Core 3.0), встроен в рантайм, быстрый, экономный по памяти, умеет source generators для AOT и скорости. Сознательно более строгий и менее «магический», чем Newtonsoft.
- **`Newtonsoft.Json` (Json.NET)** — многолетний де-факто стандарт до STJ: максимально гибкий, прощающий, с огромным числом настроек, конвертеров и поддержкой нестандартных сценариев. Всё ещё широко используется.

В Go этому противопоставлен **один** `encoding/json` из стандартной библиотеки. По духу он ближе к **STJ**: дефолтный, быстрый-достаточно, строгий, без избыточной магии. Аналога «гибкого тяжеловеса» Newtonsoft в stdlib нет — когда нужна экстремальная скорость или особые возможности, берут сторонние библиотеки (`json-iterator/go`, `bytedance/sonic`), но это не про гибкость, а про производительность.

> Практический вывод: переходя с .NET, мысленно сопоставляйте `encoding/json` с `System.Text.Json`, а не с Newtonsoft. Тогда меньше сюрпризов: оба строгие, оба заточены под производительность, оба требуют явных усилий для «нестандартного».

## Атрибуты против тегов

Самое заметное различие на поверхности кода. В .NET поведение сериализации задаётся **атрибутами** на типе/свойстве; в Go — **строковыми тегами** на поле структуры.

```csharp
// .NET (System.Text.Json)
public class Product
{
    [JsonPropertyName("id")]
    public int Id { get; set; }

    [JsonIgnore]
    public string Internal { get; set; }

    [JsonIgnore(Condition = JsonIgnoreCondition.WhenWritingDefault)]
    public double Price { get; set; }
}
```

```go
// Go (encoding/json)
type Product struct {
    ID       int     `json:"id"`
    Internal string  `json:"-"`
    Price    float64 `json:"price,omitempty"`
}
```

Принципиальная разница в природе механизма: атрибут — это типизированная аннотация, проверяемая компилятором; тег — это **обычная строка**, которую `encoding/json` парсит в рантайме. Опечатка в теге (`json:"naem"`) не вызовет ошибки компиляции — поле просто получит не то имя. (Линтеры вроде `go vet` ловят грубые ошибки синтаксиса тегов, но не семантику.) Зато теги — это часть значения структуры, их видно прямо у поля, и они работают для нескольких форматов сразу (`json:"x" yaml:"x"`).

## Регистр имён: главная ловушка миграции

Здесь модели расходятся сильнее всего, и это источник реальных багов при переходе.

**System.Text.Json по умолчанию матчит имена при десериализации регистронезависимо** (`PropertyNameCaseInsensitive = true` в дефолтных веб-настройках ASP.NET; в «голом» `JsonSerializer` дефолт строгий — нюанс, который путает и самих .NET-разработчиков). На запись STJ использует имя свойства как есть либо применяет `PropertyNamingPolicy` (например, `CamelCase`).

**Go матчит по тегу/имени поля: сначала точное совпадение, при его отсутствии — регистронезависимый фолбэк.** Но на **запись** имя строго детерминировано тегом или именем поля — никакой «политики именования» на весь сериализатор нет. Если хотите `camelCase` или `snake_case` в JSON — это пишется тегом на **каждом** поле, а не одной настройкой:

```go
type User struct {
    FirstName string `json:"firstName"` // camelCase задаётся вручную, тегом
    LastName  string `json:"lastName"`
}
```

Это, пожалуй, самое раздражающее для выходца из .NET: в STJ один `PropertyNamingPolicy = JsonNamingPolicy.CamelCase` переименует все свойства разом, а в Go придётся проставить тег у каждого поля. Глобальной политики именования в `encoding/json` нет. (Кодогенераторы и линтеры частично автоматизируют расстановку тегов, но «из коробки» — руками.)

| | `System.Text.Json` | `encoding/json` |
| --- | --- | --- |
| Матчинг на чтение | регистронезависимый (в веб-дефолте) | точный, затем регистронезависимый фолбэк |
| Имя на запись | имя свойства или `PropertyNamingPolicy` | строго из тега/имени поля |
| Глобальная политика имён | да (`CamelCase` и др. одной настройкой) | **нет** — тег на каждое поле |

## Конвертеры против `Marshaler`

Кастомную (де)сериализацию обе платформы поддерживают, но по-разному её «прикрепляют».

```csharp
// .NET: отдельный класс-конвертер, регистрируется или вешается атрибутом
public class MoneyConverter : JsonConverter<Money> { /* Read/Write */ }

[JsonConverter(typeof(MoneyConverter))]
public struct Money { /* ... */ }
```

```go
// Go: методы на самом типе
func (m Money) MarshalJSON() ([]byte, error) { /* ... */ }
func (m *Money) UnmarshalJSON(data []byte) error { /* ... */ }
```

Разница концептуальная: в .NET конвертер — **внешняя сущность**, которую можно применить к типу, в том числе к **чужому** (из сторонней библиотеки), просто зарегистрировав её в `JsonSerializerOptions.Converters`. В Go контракт реализует **сам тип** своими методами — элегантно для своих типов, но для **чужого** типа так не получится: чтобы кастомизировать сериализацию типа из другой библиотеки, его придётся обернуть в свой тип-обёртку с нужными методами. Это прямое следствие того, что в Go нельзя добавлять методы к типам из чужих пакетов.

| | `System.Text.Json` | `encoding/json` |
| --- | --- | --- |
| Где живёт логика | внешний `JsonConverter<T>` | методы `MarshalJSON`/`UnmarshalJSON` на типе |
| Применение к чужому типу | да (регистрация конвертера) | **нет** (нужна обёртка) |
| Универсальное текстовое представление | `TypeConverter` / `IParsable<T>` | `encoding.TextMarshaler`/`TextUnmarshaler` |

## Полиморфизм и `$type`

Это область, где .NET заметно богаче «из коробки». Современный STJ поддерживает полиморфную (де)сериализацию декларативно:

```csharp
[JsonPolymorphic(TypeDiscriminatorPropertyName = "$type")]
[JsonDerivedType(typeof(Dog), "dog")]
[JsonDerivedType(typeof(Cat), "cat")]
public abstract class Animal { }
// сериализуется с дискриминатором "$type":"dog" и читается обратно в нужный подтип
```

Newtonsoft делает то же через `TypeNameHandling` (исторически — с известными проблемами безопасности при `TypeNameHandling.All`).

**В Go встроенного полиморфизма нет.** Идиоматичное решение — то, что разбиралось в главе про JSON: `json.RawMessage` + поле-дискриминатор + ручной `switch`:

```go
type Envelope struct {
    Type    string          `json:"type"`
    Payload json.RawMessage `json:"payload"`
}
// читаем Type, затем json.Unmarshal(env.Payload, &конкретныйТип) по switch
```

Это больше ручной работы, но и больше явности (и без класса уязвимостей, который дал `TypeNameHandling.All` в Newtonsoft, где злоумышленник мог заставить десериализатор инстанцировать произвольный тип). Если полиморфизма много, его обычно инкапсулируют в кастомный `UnmarshalJSON` на типе-обёртке.

## Производительность и роль рефлексии

Обе экосистемы прошли похожий путь, и сравнение поучительно.

- **`encoding/json` — рефлексия в рантайме.** Гибко (работает с любым типом без подготовки), но небесплатно: обход полей и тегов через `reflect` медленнее прямого кода и аллоцирует. Это осознанный дефолт Go: простота важнее последней капли скорости.
- **`System.Text.Json` — рефлексия ИЛИ source generators.** По умолчанию STJ тоже рефлексит, но умеет **компайл-тайм кодогенерацию** (`[JsonSerializable]` + `JsonSerializerContext`): Roslyn порождает код (де)сериализации на этапе сборки, убирая рантайм-рефлексию, ускоряя старт и обеспечивая AOT-совместимость.

То есть у STJ режим «без рефлексии» встроен в платформу и включается атрибутом. В Go аналог — **внешние** кодогенераторы (`easyjson` и др.), которые порождают `MarshalJSON`/`UnmarshalJSON` для ваших типов. Концепция идентична — «обменять билд-время и сгенерированный код на рантайм-скорость», — но механизм разный: Roslyn внутри компиляции у .NET против отдельного инструмента, пишущего `.go`-файлы, у Go. Это сквозная философия Go «кодогенерация вместо рантайм-магии», которой посвящён [Раздел 7](../07-code-generation/README.md).

| | `System.Text.Json` | `encoding/json` |
| --- | --- | --- |
| Дефолтный движок | рефлексия | рефлексия |
| Режим без рефлексии | source generators (встроены, атрибут) | внешние кодогенераторы (`easyjson`) |
| Механизм генерации | Roslyn в составе компиляции | отдельный инструмент → `.go`-файлы |
| Сторонние «ускорители» | редко нужны | `json-iterator`, `sonic` (drop-in) |

## Сводная таблица соответствий

| Концепция / задача | .NET | Go |
| --- | --- | --- |
| Дефолтный JSON-сериализатор | `System.Text.Json` (`JsonSerializer`) | `encoding/json` |
| Сериализовать | `JsonSerializer.Serialize(obj)` → `string` | `json.Marshal(obj)` → `[]byte` |
| Десериализовать | `JsonSerializer.Deserialize<T>(s)` | `json.Unmarshal(data, &v)` |
| С отступами | `WriteIndented = true` | `json.MarshalIndent` |
| Переименовать поле | `[JsonPropertyName("x")]` | тег `json:"x"` |
| Игнорировать поле | `[JsonIgnore]` | тег `json:"-"` |
| Опустить пустое | `JsonIgnoreCondition.WhenWritingDefault` | `omitempty` / `omitzero` (Go 1.24) |
| «Опциональное» поле (отличить отсутствие от нуля) | `int?` / `Nullable<T>` | указатель `*int` |
| Кастомная сериализация | `JsonConverter<T>` (внешний) | `MarshalJSON`/`UnmarshalJSON` (на типе) |
| Универсальное текст-представление | `TypeConverter` / `IParsable<T>` | `encoding.TextMarshaler` |
| Отложенный/сырой узел | `JsonElement` / `JsonNode` | `json.RawMessage` |
| Полиморфизм | `[JsonDerivedType]` / `$type` (встроено) | вручную: `RawMessage` + дискриминатор |
| Строгий разбор (неизвестные поля) | `UnmappedMemberHandling.Disallow` | `Decoder.DisallowUnknownFields()` |
| Потоковая запись/чтение | `Utf8JsonWriter`/`Reader`, `...Async(stream)` | `json.NewEncoder/Decoder` (поверх `io`) |
| Кодогенерация для скорости/AOT | source generators (`JsonSerializerContext`) | `easyjson` и др. (Раздел 7) |
| Глобальная политика имён | `PropertyNamingPolicy = CamelCase` | **нет** — тег на каждое поле |
| YAML | `YamlDotNet` (NuGet) | `go.yaml.in/yaml/v3` (внешний) |
| Бинарь «для своих» | `MessagePack` (совр.); `BinaryFormatter` (устарел) | `encoding/gob` |
| Кросс-языковой бинарь | `Google.Protobuf` + `protoc` | `google.golang.org/protobuf` + `protoc-gen-go` |

## «Как мне сделать привычное X»

- **«Хочу `camelCase` во всём JSON»** → в Go нет глобальной настройки; проставьте тег `json:"fieldName"` каждому полю (или сгенерируйте теги инструментом).
- **«Хочу отличить `null`/отсутствие от значения по умолчанию»** → используйте указатель (`*int`, `*bool`, `*string`) — это аналог `int?`. `nil` = «не задано».
- **«У меня свойство типа `decimal`/деньги/особое время»** → реализуйте `MarshalJSON`/`UnmarshalJSON` на типе (вместо `JsonConverter`); для простого строкового вида — `TextMarshaler`.
- **«Мне нужна кастомная сериализация чужого типа»** → оберните его в свой тип с нужными методами (методы к чужим типам в Go добавить нельзя).
- **«У меня иерархия классов (полиморфизм)»** → в Go руками: `json.RawMessage` + поле-дискриминатор + `switch`; инкапсулируйте в `UnmarshalJSON`.
- **«Хочу падать на неизвестных полях (строгий конфиг)»** → `Decoder.DisallowUnknownFields()` (JSON) или `Decoder.KnownFields(true)` (YAML).
- **«У меня тормозит JSON на горячем пути»** → сперва профилируйте; затем либо drop-in `json-iterator`/`sonic`, либо кодогенерация `easyjson` (аналог STJ source-gen).
- **«Мне нужен бинарный обмен между Go-сервисами»** → `encoding/gob` (если кросс-язычность не нужна) или protobuf (если нужна или планируется).
- **«У меня уже есть `.proto` из .NET-проекта»** → используйте его как есть: protobuf-схема идентична, сгенерируйте Go-код через `protoc`/`buf`.

## Итог

- Сопоставляйте `encoding/json` со **`System.Text.Json`** (строгий, быстрый дефолт), а не с гибким Newtonsoft; аналога «тяжёлого гибкого» сериализатора в stdlib Go нет.
- **Атрибуты → теги**: поведение задаётся строковыми тегами поля, проверяемыми в рантайме, а не типизированными атрибутами; зато один набор тегов работает на несколько форматов.
- **Главная ловушка миграции — регистр имён**: в Go нет глобальной `PropertyNamingPolicy`, `camelCase`/`snake_case` проставляется тегом на каждое поле; матчинг на чтение — точный с регистронезависимым фолбэком.
- **Конвертеры → методы на типе**: `MarshalJSON`/`Unmarshaler` живут на самом типе (не внешний `JsonConverter`), поэтому для чужого типа нужна обёртка. Полиморфизм в Go не встроен — `RawMessage` + дискриминатор вручную.
- **Перформанс и рефлексия**: оба по умолчанию рефлексят; путь «без рефлексии» у STJ — встроенные source generators, у Go — внешние кодогенераторы (мостик к Разделу 7). YAML и protobuf в обоих мирах — внешние зависимости, причём protobuf полностью совместим между .NET и Go.

На этом раздел о сериализации завершён. Логичное продолжение — [Раздел 7: Кодогенерация как философия языка](../07-code-generation/README.md), где «генерация вместо рефлексии», которую мы упоминали для JSON и protobuf, разбирается как сквозной принцип Go.
