# JSON (`encoding/json`)

JSON в Go живёт в стандартной библиотеке — пакет `encoding/json`. Это ваш аналог `System.Text.Json`: дефолтный, всегда под рукой, без сторонних зависимостей. Идейно он близок к STJ, но управляется не атрибутами на типе, а **тегами полей структуры**, и подчиняется одному жёсткому правилу, на котором спотыкаются все новички: **сериализуются только экспортируемые (с заглавной буквы) поля**. Эта глава разбирает пакет вглубь — от базовых `Marshal`/`Unmarshal` до кастомных маршалеров, потоковых декодеров и тонкостей с `null`.

## `Marshal` и `Unmarshal`: две функции, с которых всё начинается

Весь базовый API — это две функции-зеркала:

```go
func Marshal(v any) ([]byte, error)       // объект → JSON
func Unmarshal(data []byte, v any) error  // JSON → объект (через указатель!)
```

```go
type User struct {
    Name string
    Age  int
}

// Marshal: структура → []byte
u := User{Name: "Alice", Age: 30}
data, err := json.Marshal(u)
// data == `{"Name":"Alice","Age":30}`

// Unmarshal: []byte → структура (обязательно через указатель)
var back User
err = json.Unmarshal(data, &back)
// back == User{Name:"Alice", Age:30}
```

Два момента сразу бросаются в глаза .NET-разработчику:

1. **`Marshal` возвращает `[]byte`, а не `string`.** JSON в Go — это байты (UTF-8), а не строка. Если нужна строка — `string(data)`. Это типично для Go: I/O оперирует байтами.
2. **`Unmarshal` принимает указатель `&back`.** Иначе функция получила бы копию (Раздел 1, семантика значений) и не смогла бы заполнить вашу переменную. Забыть `&` — классическая ошибка; компилятор её не ловит, но `Unmarshal` вернёт ошибку «non-pointer».

> **Параллель с .NET:** `json.Marshal(u)` ≈ `JsonSerializer.Serialize(u)`, `json.Unmarshal(data, &u)` ≈ `JsonSerializer.Deserialize<User>(data)`. Разница: STJ возвращает `string` (или пишет в `Utf8JsonWriter`), а Go всегда отдаёт `[]byte`; и в Go вы явно передаёте указатель на цель десериализации, тогда как STJ возвращает результат значением через дженерик-параметр.

Для читаемого вывода с отступами есть `json.MarshalIndent`:

```go
data, _ := json.MarshalIndent(u, "", "  ") // префикс строки, отступ
// {
//   "Name": "Alice",
//   "Age": 30
// }
```

Это аналог `new JsonSerializerOptions { WriteIndented = true }`.

## Главное правило: только экспортируемые поля

Это самая частая ловушка для новичка. `encoding/json` использует рефлексию, а рефлексия в Go **не видит неэкспортируемые (строчные) поля** — она не может ни прочитать, ни записать их. Поэтому поле с маленькой буквы просто **молча исчезает** из JSON:

```go
type Account struct {
    ID      int    // экспортируемое → попадёт в JSON
    balance int    // строчное → НЕ попадёт, тихо игнорируется ❌
}

a := Account{ID: 1, balance: 100}
data, _ := json.Marshal(a)
// {"ID":1} — balance пропал, без ошибки и предупреждения
```

В .NET вы привыкли к обратному: `System.Text.Json` по умолчанию сериализует **публичные свойства**, а приватные поля и так не сериализуются — но там это вопрос выбора политики (можно включить поля, можно приватные через атрибуты). В Go это **жёсткое ограничение языка**: рефлексия физически не имеет доступа к строчным полям из другого пакета, и `encoding/json` — не исключение. Хотите, чтобы поле сериализовалось, — делайте его экспортируемым, а нужное JSON-имя задавайте тегом.

> **Параллель с .NET:** в C# вы бы написали `public int Balance { get; private set; }` — публичное для сериализатора, но с защищённой записью. В Go такого «публичное для рефлексии, приватное для кода» нет: либо `Balance` (видно всем, включая JSON), либо `balance` (не видно ни JSON, ни внешним пакетам). Инкапсуляцию задают на уровне пакета, а не поля.

## Теги структур: переименование, пропуск, опции

Тег — это строка в обратных кавычках после поля. Для JSON ключ тега — `json`:

```go
type Product struct {
    ID        int     `json:"id"`                   // переименование в "id"
    Name      string  `json:"name"`
    Price     float64 `json:"price,omitempty"`      // опустить, если пусто
    Internal  string  `json:"-"`                     // никогда не сериализовать
    SKU       string  `json:"sku,omitempty"`
    legacyKey string                                  // строчное — игнор в любом случае
}
```

Синтаксис тега: `json:"<имя>,<опция1>,<опция2>"`. Разберём части.

### Переименование и пропуск

- `json:"id"` — в JSON поле будет называться `id`, а не `ID`. Это прямой аналог `[JsonPropertyName("id")]`.
- `json:"-"` — поле **полностью исключается** из (де)сериализации. Аналог `[JsonIgnore]`. Тонкость: если вам нужно поле с буквальным именем `"-"`, используйте `json:"-,"` (с запятой).
- `json:",omitempty"` (имя пустое, только опция) — поле сохраняет своё Go-имя `Field`, но применяется `omitempty`. Пустое имя до запятой = «оставь имя как есть».

### Матчинг при десериализации: точный, но регистронезависимый

При `Unmarshal` Go сначала ищет поле, чьё JSON-имя (из тега или имени поля) **точно** совпадает с ключом. Если точного совпадения нет — выполняет **регистронезависимый** поиск. То есть ключ `"name"` найдёт поле с тегом `json:"name"`, а при отсутствии тега — поле `Name`. Это важное отличие от записи: при `Marshal` имя берётся строго из тега/имени поля, регистр фиксирован.

> **Параллель с .NET:** `System.Text.Json` по умолчанию матчит имена **регистронезависимо** при десериализации (`PropertyNameCaseInsensitive`), и это поведение в Go частично совпадает — но в Go точное совпадение всегда в приоритете, а регистронезависимость — лишь фолбэк. На запись Go, как и STJ с `PropertyNamingPolicy`, использует ровно то имя, что задано тегом.

## `omitempty` и его подводные камни

`omitempty` опускает поле из вывода, если его значение «пустое». Звучит просто, но определение «пустого» в Go узкое и **не совпадает** с интуицией. Поле опускается, только если значение равно:

- `false` для `bool`;
- `0` для числовых типов;
- `""` для строк;
- `nil` для указателя, интерфейса, **слайса**, **мапы**, канала, функции;
- слайс или мапа **нулевой длины** (len == 0).

А вот чего `omitempty` **НЕ** считает пустым — и это главная ловушка:

```go
type Config struct {
    Timeout  time.Duration `json:"timeout,omitempty"`  // 0 → опустится ✅
    Endpoint Address       `json:"endpoint,omitempty"` // пустая СТРУКТУРА → НЕ опустится ❌
    Count    int           `json:"count,omitempty"`    // 0 → опустится, даже если 0 осмыслен ❌
}

type Address struct{ Host string; Port int }
```

Две беды:

1. **`omitempty` не работает на структурах.** Пустая `struct{}` (все поля нулевые) — это **не** «пустое значение» в смысле `omitempty`. Поле `endpoint` сериализуется как `{"host":"","port":0}`, хотя вы ждали его исчезновения. Причина: у структуры нет понятия «нулевая длина», а сравнения «структура == свой ноль» `omitempty` не делает.

2. **`omitempty` не отличает «не задано» от «осмысленный ноль».** Если `0` или `""` — валидное значение (счётчик, координата, флаг «выключено»), `omitempty` всё равно его выкинет. Получатель не увидит разницы между «поля не было» и «поле равнялось нулю».

**Классические лекарства** до Go 1.24:

```go
// Указатель: nil = «не задано», &0 = «явный ноль»
type Patch struct {
    Active *bool `json:"active,omitempty"` // nil опустится; *false — нет
    Count  *int  `json:"count,omitempty"`  // позволяет различить отсутствие и 0
}
```

Указатель решает обе проблемы: `nil` опускается, а `&value` (даже на нулевое значение) — нет. Цена — возня с разыменованием и аллокациями. Для структур помогает либо указатель на структуру (`*Address`), либо кастомный `MarshalJSON`.

> **Параллель с .NET:** `omitempty` ≈ `JsonIgnoreCondition.WhenWritingDefault` (опускать, если значение по умолчанию). Но STJ для структур-`struct` ведёт себя аналогично проблемно, поэтому в .NET для «опциональных» полей используют `Nullable<T>` (`int?`, `bool?`) — это **прямой аналог** Go-указателя `*int`/`*bool`: `null` ↔ «не задано», значение ↔ «задано». Привычка из C# «делаю поле `int?`, чтобы отличить 0 от отсутствия» один-в-один переносится в Go как `*int`.

## `omitzero` (Go 1.24): то, чем `omitempty` должен был быть

Go 1.24 добавил опцию `omitzero`, которая чинит главные недостатки `omitempty`. Логика «пустоты» теперь совпадает с языковым понятием **нулевого значения**:

- если у типа есть метод `IsZero() bool` — используется он (например, у `time.Time` есть `IsZero()`, и `omitzero` наконец-то корректно опускает «нулевое время»);
- иначе значение считается нулевым, если оно равно zero value своего типа — **включая структуры** (пустая `struct{}` со всеми нулевыми полями опустится).

```go
type Event struct {
    At      time.Time `json:"at,omitzero"`      // нулевое время ОПУСТИТСЯ ✅ (omitempty не умел)
    Source  Address   `json:"source,omitzero"`  // пустая структура ОПУСТИТСЯ ✅
    Retries int       `json:"retries,omitzero"` // 0 опустится
}
```

Важные различия `omitempty` vs `omitzero`:

| | `omitempty` | `omitzero` (Go 1.24+) |
| --- | --- | --- |
| `0`, `""`, `false` | опускает | опускает |
| `nil` указатель/слайс/мапа | опускает | опускает |
| Слайс/мапа `len==0`, но не `nil` (`[]T{}`) | **опускает** | **НЕ опускает** (не nil → не ноль) |
| Пустая структура | НЕ опускает ❌ | **опускает** ✅ |
| `time.Time{}` (нулевое время) | НЕ опускает ❌ | **опускает** ✅ (через `IsZero()`) |

Практическая рекомендация сообщества: **для слайсов и мап оставляйте `omitempty`** (обычно вы хотите выкидывать и пустой `[]`, не только `nil`), а для **всего остального — `omitzero`**, особенно для структур и `time.Time`. Можно комбинировать обе опции на одном поле (`json:"x,omitempty,omitzero"`), но это редкий случай.

> **Параллель с .NET:** появление `omitzero` приблизило Go к гибкости STJ, где условия пропуска (`JsonIgnoreCondition`) и `IsZero`-подобная логика давно настраиваемы. По сути `omitzero` — это «опускать, если `EqualityComparer<T>.Default.Equals(value, default)`», с уважением к пользовательскому `IsZero()`.

## Опция `string`: число/булево как JSON-строка

Опция `string` в теге заставляет поле кодироваться/декодироваться как JSON-**строку**, хотя в Go это число, булево или строка. Нужна для интеропа с API, где числа приходят в кавычках (частый случай — большие int64, которые JavaScript теряет в точности):

```go
type Money struct {
    Cents int64 `json:"cents,string"` // в JSON будет "12345", а не 12345
}

m := Money{Cents: 12345}
data, _ := json.Marshal(m)
// {"cents":"12345"} — значение в кавычках
```

Применима только к строкам, булевым и числовым типам. Это нишевый инструмент — не путайте с обычным переименованием.

## `null`, нулевые значения и указатели для «опциональных» полей

Это зона, где модель Go (нулевые значения вместо `null`, Раздел 2) встречается с JSON, где `null` существует. Разберём по случаям.

### Десериализация `null`

JSON `null` при `Unmarshal` обрабатывается так: для **указателя** поле становится `nil`; для **не-указателя** поле остаётся **без изменений** (сохраняет текущее значение, обычно нулевое). То есть `null` не «обнуляет» не-указательное поле — он его просто не трогает:

```go
type T struct {
    A *int `json:"a"`
    B int  `json:"b"`
}
var t T
json.Unmarshal([]byte(`{"a":null,"b":null}`), &t)
// t.A == nil (указатель занулён)
// t.B == 0 (не тронут; остался zero value)
```

### Сериализация: nil-указатель → `null`

При `Marshal` `nil`-указатель даёт `null` (если нет `omitempty`):

```go
type T struct{ A *int `json:"a"` }
json.Marshal(T{A: nil}) // {"a":null}
```

Это и есть способ осознанно вывести `null` в JSON — через `nil`-указатель без `omitempty`. Если же `omitempty` стоит, `nil` приведёт к пропуску поля вместо `null`. Выбор «`null` или отсутствие ключа» — это выбор «без `omitempty` / с `omitempty`».

### Коварная разница: nil-слайс → `null`, пустой слайс → `[]`

Очень частый источник багов в API. **nil-слайс** и **пустой непустой слайс** сериализуются по-разному:

```go
type Resp struct {
    Items []string `json:"items"`
}

json.Marshal(Resp{Items: nil})         // {"items":null}  ← nil-слайс
json.Marshal(Resp{Items: []string{}})  // {"items":[]}    ← инициализированный пустой
```

`nil`-слайс (объявлен, но не инициализирован) даёт `null`, а слайс, созданный как `[]string{}` или `make([]string, 0)`, даёт `[]`. Для потребителей API (особенно JS-фронтенда) `null` и `[]` — это две большие разницы: на `null` нельзя вызвать `.map()`, а на `[]` — можно. Идиома: если контракт требует всегда массив, инициализируйте слайс пустым, а не оставляйте `nil`:

```go
items := []string{} // не var items []string — чтобы получить [], а не null
```

Это, кстати, ещё одна причина, почему для слайсов оставляют `omitempty` (а не `omitzero`): `omitempty` выкинет и `nil`, и `[]` одинаково, избавив от вопроса вовсе.

> **Параллель с .NET:** в C# `List<string>` со значением `null` сериализуется в `null`, а пустой `new List<string>()` — в `[]` — ровно та же дихотомия. Привычка инициализировать коллекции пустыми (`= new()`), чтобы не отдавать `null` в API, прямо переносится в Go.

### Числа в `any` → всегда `float64` (классический сюрприз)

Если вы декодируете JSON в `any` (или `map[string]any`) — например, при разборе произвольной структуры — все числа становятся `float64`, **независимо** от того, были они целыми или дробными:

```go
var data map[string]any
json.Unmarshal([]byte(`{"id":42}`), &data)

id := data["id"]          // тип — float64, не int!
n := id.(int)             // ПАНИКА: interface conversion, это float64 ❌
n := int(id.(float64))    // правильно: сначала float64, потом конверсия ✅
```

Причина: у JSON один числовой тип, и `encoding/json`, не зная целевого типа, выбирает самый общий — `float64`. Это та же история, что боксинг в `object` в C#, но с фиксацией именно на `double`. Последствия: для больших `int64` `float64` теряет точность (после 2^53), и `42` извлекается как `float64(42)`, а не `int`.

**Лекарство** — `Decoder.UseNumber()`, который складывает числа в тип `json.Number` (строка с методами `.Int64()`/`.Float64()`), сохраняя точность:

```go
dec := json.NewDecoder(strings.NewReader(`{"id":9007199254740993}`))
dec.UseNumber()
var data map[string]any
dec.Decode(&data)
n, _ := data["id"].(json.Number).Int64() // точное int64 ✅
```

Но лучший совет — **не декодируйте в `any`, если знаете структуру**. Опишите `struct` с нужными типами полей (`int64`, `int`), и `encoding/json` сам сконвертирует число в этот тип корректно. `any` — крайнее средство для действительно динамического JSON.

> **Параллель с .NET:** в `System.Text.Json` десериализация в `object`/`JsonElement` тоже не даёт готовый `int` — вы получаете `JsonElement` с `ValueKind: Number` и вызываете `.GetInt32()`/`.GetInt64()`/`.GetDouble()`. Так что «число в динамическом контейнере требует явной конверсии» — общая боль обоих миров; Go просто фиксирует промежуточный тип как `float64`, а STJ — как `JsonElement`.

## Кастомная сериализация: `json.Marshaler` / `Unmarshaler`

Когда дефолтного маппинга мало (особый формат времени, деньги, перечисления, обёртки), тип реализует интерфейсы:

```go
type Marshaler interface {
    MarshalJSON() ([]byte, error)
}
type Unmarshaler interface {
    UnmarshalJSON([]byte) error
}
```

`encoding/json` проверяет, реализует ли тип эти интерфейсы, и если да — вызывает ваши методы вместо рефлексии. Это **главное отличие от .NET-конвертеров**: контракт реализует **сам тип**, а не внешний класс-конвертер.

Пример — деньги как строка «12.34» вовне, `int64` центов внутри:

```go
type Money int64 // храним центы

func (m Money) MarshalJSON() ([]byte, error) {
    return json.Marshal(fmt.Sprintf("%d.%02d", m/100, m%100))
}

func (m *Money) UnmarshalJSON(data []byte) error {
    var s string
    if err := json.Unmarshal(data, &s); err != nil {
        return err
    }
    var dollars, cents int64
    if _, err := fmt.Sscanf(s, "%d.%d", &dollars, &cents); err != nil {
        return err
    }
    *m = Money(dollars*100 + cents)
    return nil
}
```

```go
type Order struct {
    Total Money `json:"total"`
}
data, _ := json.Marshal(Order{Total: 1234})
// {"total":"12.34"} — наш формат ✅
```

Обратите внимание на ресиверы: `MarshalJSON` — на значении (`m Money`), `UnmarshalJSON` — на **указателе** (`*Money`), потому что он должен изменить значение (Раздел 2, pointer receiver для мутации). Это типовая раскладка.

### `encoding.TextMarshaler` / `TextUnmarshaler`

Часто проще реализовать «текстовую» сериализацию — пакет `encoding` определяет:

```go
type TextMarshaler interface{ MarshalText() ([]byte, error) }
type TextUnmarshaler interface{ UnmarshalText(text []byte) error }
```

`encoding/json` уважает эти интерфейсы (если нет `MarshalJSON`): значение оборачивается в JSON-строку автоматически. Бонус: тот же `TextMarshaler` подхватывают и `yaml.v3`, и ключи мап, и `encoding/xml` — реализуете один раз, работает везде. Идеально для перечислений:

```go
type Status int

const (
    Active Status = iota
    Banned
)

func (s Status) MarshalText() ([]byte, error) {
    switch s {
    case Active:
        return []byte("active"), nil
    case Banned:
        return []byte("banned"), nil
    }
    return nil, fmt.Errorf("unknown status %d", s)
}
// В JSON Status сериализуется как "active"/"banned" — и так же в YAML.
```

> **Параллель с .NET:** `json.Marshaler`/`Unmarshaler` ≈ `JsonConverter<T>` из `System.Text.Json` (или `JsonConverter` из Newtonsoft), но с инверсией: в .NET конвертер — это **отдельный класс**, который вы регистрируете в `JsonSerializerOptions.Converters` или вешаете атрибутом `[JsonConverter(typeof(MoneyConverter))]`. В Go логика (де)сериализации живёт **на самом типе** как метод — нет регистрации, нет отдельного класса. Минус Go-подхода: нельзя задать кастомную сериализацию для **чужого** типа (из другой библиотеки), не обернув его в свой; в .NET внешний конвертер это позволяет. `TextMarshaler` ≈ `TypeConverter` / `IParsable<T>` — «универсальное строковое представление».

## `json.RawMessage`: отложенный разбор

`json.RawMessage` — это `[]byte`, которые `encoding/json` **не разбирает**, а сохраняет как есть (при `Unmarshal`) или вставляет дословно (при `Marshal`). Это инструмент «разберу позже» — например, когда часть документа зависит от поля-дискриминатора:

```go
type Envelope struct {
    Type    string          `json:"type"`
    Payload json.RawMessage `json:"payload"` // НЕ разбирается сразу
}

func handle(data []byte) error {
    var env Envelope
    if err := json.Unmarshal(data, &env); err != nil {
        return err
    }
    switch env.Type { // разбираем payload по типу — отложенно
    case "user":
        var u User
        return json.Unmarshal(env.Payload, &u)
    case "order":
        var o Order
        return json.Unmarshal(env.Payload, &o)
    }
    return fmt.Errorf("unknown type %q", env.Type)
}
```

Это базовый способ сделать в Go **полиморфную десериализацию** (которой нет из коробки): сначала достаём дискриминатор, потом по нему разбираем «сырой» кусок в нужный тип. `RawMessage` также избавляет от двойного разбора и сохраняет исходное форматирование значения.

> **Параллель с .NET:** `json.RawMessage` ≈ `JsonElement` (отложенный, навигируемый узел) или `JsonNode`. Сценарий «дискриминатор + сырой payload» в STJ закрывается встроенным полиморфизмом через `[JsonPolymorphic]`/`[JsonDerivedType]` и `$type`. В Go встроенного полиморфизма нет — `RawMessage` + ручной `switch` по типу и есть идиоматичное решение.

## Потоковые `Encoder` / `Decoder`

`Marshal`/`Unmarshal` работают с `[]byte` целиком в памяти. Для потоков (HTTP-тела, файлы, NDJSON, большие данные) есть `json.NewEncoder(w io.Writer)` и `json.NewDecoder(r io.Reader)` — они пишут/читают прямо из `io`-интерфейсов:

```go
// Запись прямо в ответ HTTP без промежуточного буфера
func handler(w http.ResponseWriter, r *http.Request) {
    user := User{Name: "Alice", Age: 30}
    w.Header().Set("Content-Type", "application/json")
    json.NewEncoder(w).Encode(user) // пишет в w напрямую
}

// Чтение из тела запроса
func create(w http.ResponseWriter, r *http.Request) {
    var u User
    if err := json.NewDecoder(r.Body).Decode(&u); err != nil {
        http.Error(w, err.Error(), http.StatusBadRequest)
        return
    }
    // ...
}
```

`Decoder` ещё и умеет читать **поток из нескольких JSON-значений** подряд (вызывая `Decode` в цикле) — удобно для NDJSON/JSON Lines.

### `DisallowUnknownFields`: строгий разбор

По умолчанию `Unmarshal` **молча игнорирует** ключи JSON, которым нет соответствия в структуре. Это часто нежелательно (опечатка в конфиге пройдёт незамеченной). `Decoder.DisallowUnknownFields()` включает строгий режим — неизвестный ключ вызовет ошибку:

```go
dec := json.NewDecoder(r.Body)
dec.DisallowUnknownFields()
var cfg Config
if err := dec.Decode(&cfg); err != nil {
    // ошибка вида: json: unknown field "tytmeout" (опечатка поймана) ✅
}
```

Это аналог `JsonSerializerOptions { UnmappedMemberHandling = JsonUnmappedMemberHandling.Disallow }` в современном STJ (или `MissingMemberHandling.Error` в Newtonsoft). Полезно для конфигов и строгих контрактов; для «прощающих» публичных API обычно оставляют дефолт.

> **Параллель с .NET:** `json.NewEncoder(w).Encode(v)` ≈ `JsonSerializer.Serialize(utf8JsonWriter, v)` / `SerializeAsync(stream, v)`; `json.NewDecoder(r).Decode(&v)` ≈ `JsonSerializer.DeserializeAsync<T>(stream)`. Ключевая разница в духе языка: в Go это **синхронный** код поверх `io.Reader`/`io.Writer` — никакого `async`/`await` (горутины и блокировка дёшевы, Раздел 3), тогда как в .NET потоковая работа почти всегда асинхронна (`...Async`).

## Производительность и рефлексия: мостик к кодогенерации

`encoding/json` под капотом — это **рефлексия в рантайме**: на каждый вызов он обходит поля типа, читает теги, проверяет реализацию интерфейсов. Это удобно (работает с любым типом без подготовки), но не бесплатно — рефлексия медленнее прямого кода и аллоцирует. Для горячих путей (высоконагруженные сервисы, парсинг гигабайтов) это бывает узким местом.

Что с этим делают в экосистеме Go:

- **Сторонние рантайм-библиотеки**, совместимые по API: `github.com/json-iterator/go`, `github.com/bytedance/sonic` (последняя использует JIT/SIMD) — заметно быстрее stdlib, drop-in замена.
- **Кодогенерация**: инструменты вроде `easyjson` или `ffjson` генерируют для ваших типов специализированные `MarshalJSON`/`UnmarshalJSON` **на этапе сборки**, без рефлексии в рантайме. Это путь «обменять немного билд-времени и сгенерированного кода на скорость».

Кодогенерация — глубоко идиоматичный для Go способ убрать рантайм-рефлексию, и это сквозная тема языка (`stringer`, `mockery`, `protoc`). Подробно философию «генерация на компиляции вместо магии в рантайме» разбирает [Раздел 7](../07-code-generation/README.md).

> **Параллель с .NET:** ровно та же дилемма решается в .NET через **source generators** `System.Text.Json` (`[JsonSerializable]` + `JsonSerializerContext`): компилятор генерирует код (де)сериализации на этапе сборки, убирая рефлексию, ускоряя старт и делая код AOT-совместимым. Go-кодогенераторы JSON (`easyjson`) — это прямой концептуальный аналог STJ source-gen. Разница в механизме: .NET генерирует через Roslyn в рамках компиляции, Go — внешним инструментом, порождающим `.go`-файлы (детали — Раздел 7).

## Итог

- `encoding/json` — stdlib-аналог `System.Text.Json`: `Marshal`/`Unmarshal` (через указатель!), вывод — `[]byte`, а не `string`. Управление — **тегами полей**, а не атрибутами.
- **Только экспортируемые (заглавные) поля** сериализуются — это жёсткое ограничение рефлексии Go, а не настраиваемая политика. JSON-имя задаётся тегом `json:"..."`.
- `omitempty` опускает «пустые» значения, но **не работает на структурах** и **не отличает осмысленный ноль от отсутствия**; для опциональных полей берут указатели (`*int` ≈ `int?`). `omitzero` (Go 1.24) чинит это: уважает `IsZero()` и нулевые структуры/`time.Time`. Идиома: `omitempty` для слайсов/мап, `omitzero` для остального.
- `null`/нулевые: nil-указатель ↔ `null`; **nil-слайс → `null`, а `[]T{}` → `[]`** (частый баг API); числа в `any` всегда `float64` (теряют точность — берите `UseNumber()` или конкретные типы).
- Кастомная сериализация — через `json.Marshaler`/`Unmarshaler` или `encoding.TextMarshaler` **на самом типе** (vs внешний `JsonConverter` в .NET); `RawMessage` — отложенный разбор и основа ручного полиморфизма.
- Потоки — `Encoder`/`Decoder` поверх `io`; `DisallowUnknownFields()` для строгого разбора. Рефлексия в рантайме — цена удобства; ускоряют сторонние библиотеки и кодогенерация (мостик к Разделу 7).

Дальше — YAML: формат конфигов, который, в отличие от JSON, в стандартную библиотеку не входит.
