# Сравнение с .NET

Это итоговая глава раздела — консолидированный «мостик» между отказоустойчивостью в .NET и в Go. Все три паттерна (ретраи, circuit breaker, rate limiting) мы уже разобрали поштучно; теперь проговорим главное архитектурное различие двух миров, которое сильнее всего ломает интуицию .NET-разработчика: **единый фреймворк Polly против модульного, децентрализованного подхода Go**.

## Фундаментальное различие: единый фреймворк vs набор кирпичей

### Как это устроено в .NET: Polly как единый фреймворк

В .NET отказоустойчивость — это де-факто **один фреймворк**, Polly. Он настолько укоренился, что в .NET 8 вошёл в платформу под именем `Microsoft.Extensions.Resilience` / `Microsoft.Extensions.Http.Resilience`. Все стратегии живут в едином абстрактном конвейере — `ResiliencePipeline`, — который вы собираете билдером и применяете одним вызовом:

```csharp
// .NET (Polly v8): ВСЕ стратегии в одном пайплайне, единый API
ResiliencePipeline pipeline = new ResiliencePipelineBuilder()
    .AddRetry(new RetryStrategyOptions
    {
        MaxRetryAttempts = 3,
        BackoffType = DelayBackoffType.Exponential,
        UseJitter = true,
        ShouldHandle = new PredicateBuilder().Handle<HttpRequestException>(),
    })
    .AddCircuitBreaker(new CircuitBreakerStrategyOptions
    {
        FailureRatio = 0.5,
        SamplingDuration = TimeSpan.FromSeconds(30),
        BreakDuration = TimeSpan.FromSeconds(10),
    })
    .AddTimeout(TimeSpan.FromSeconds(5))
    .Build();

// Применяется одним вызовом; стратегии вложены автоматически (timeout → CB → retry → call)
await pipeline.ExecuteAsync(async ct => await httpClient.GetAsync(url, ct), cancellationToken);
```

Ключевые свойства этого подхода:

- **Единый API и словарь.** Все паттерны конфигурируются одинаково (`...StrategyOptions`), исполняются одинаково (`ExecuteAsync`), и порядок вложенности задаётся порядком вызовов билдера — фреймворк собирает «луковицу» сам.
- **Широкий каталог из коробки.** Помимо трёх наших паттернов, Polly v8 даёт `Timeout`, `Fallback`, `Hedging` (хеджирование — параллельный запуск дублей), `RateLimiter` (поверх `System.Threading.RateLimiting`), `Bulkhead`/concurrency limiter. Всё — части одного целого.
- **Высокая «открываемость» (discoverability).** Есть одно место («как у нас сделана отказоустойчивость»), одна документация, единые телеметрия и DI-интеграция (`AddResiliencePipeline`).

### Как это устроено в Go: набор независимых кирпичей

В Go **единого фреймворка отказоустойчивости нет — и это сознательно** (та же философия минимализма, что и во всём языке, см. [Раздел 1](../01-philosophy-and-syntax/README.md)). Вместо одного «комбайна» — россыпь маленьких инструментов, каждый из которых делает ровно одну вещь и ничего не знает о других:

- ретрай — цикл `for` руками или `avast/retry-go`;
- circuit breaker — `sony/gobreaker`;
- rate limiting — `golang.org/x/time/rate`;
- timeout — это вообще не «паттерн отказоустойчивости», а `context.WithTimeout` ([Раздел 3](../03-concurrency/03-select-and-context.md));
- fallback — обычный `if err != nil { ... }`, никакого примитива не нужно;
- bulkhead — буферизованный канал-семафор.

Композиция — **ваша задача**, и она явная: вы вкладываете вызовы функций друг в друга руками.

```go
// Go: НЕТ единого пайплайна. Вкладываете кирпичи сами, явно.
func resilientFetch(ctx context.Context, cb *gobreaker.CircuitBreaker[[]byte]) ([]byte, error) {
	// timeout — это context, а не «стратегия»
	ctx, cancel := context.WithTimeout(ctx, 5*time.Second)
	defer cancel()

	// breaker СНАРУЖИ, retry ВНУТРИ (см. главу 2): порядок задаёте вы, вложенностью кода
	return cb.Execute(func() ([]byte, error) {
		return retry.DoWithData(
			func() ([]byte, error) { return doGet(ctx) },
			retry.Context(ctx),
			retry.Attempts(3),
			retry.DelayType(retry.CombineDelay(retry.BackOffDelay, retry.RandomDelay)),
		)
	})
}
```

Ключевые свойства:

- **Никакого общего API.** `retry.Do`, `cb.Execute`, `lim.Wait` — три разные библиотеки с тремя разными сигнатурами и философиями. Объединяет их только то, что все они уважают `context`.
- **Порядок и склейка — явные и в вашем коде.** «Луковицу» (что снаружи, что внутри) вы выкладываете сами вложением вызовов — это и плюс (полный контроль, см. дискуссию о порядке retry/breaker в [главе 2](./02-circuit-breaker.md)), и минус (легко собрать неправильно).
- **Меньше «магии», больше прозрачности.** Видно ровно то, что написано: где какой токен берётся, где какая пауза. Нет скрытого поведения фреймворка.

> Это и есть суть различия. В .NET отказоустойчивость — **продукт** (Polly): один фреймворк, который вы конфигурируете. В Go это **сумма решений**: вы выбираете по кирпичу на задачу и складываете их сами. .NET оптимизирует под «всё из коробки и единообразно», Go — под «минимально и явно».

## Сводная таблица: задача → Polly → Go

| Задача | Стратегия Polly (v8) | Решение в Go |
| --- | --- | --- |
| Простой повтор с фикс. паузой | `AddRetry` (Constant) | цикл `for` + `select`/`ctx.Done()` (свой код) |
| Повтор с backoff + jitter | `AddRetry` (`Exponential`, `UseJitter`) | `avast/retry-go`: `BackOffDelay` + `RandomDelay`, `MaxJitter` |
| Условие «что ретраить» | `ShouldHandle` | `retry.RetryIf` / `retry.Unrecoverable` |
| Circuit breaker | `AddCircuitBreaker` (`FailureRatio`, `BreakDuration`) | `sony/gobreaker`: `Settings.ReadyToTrip`/`Timeout`, `cb.Execute` |
| Что считать провалом для CB | `ShouldHandle` | `Settings.IsSuccessful` / `IsExcluded` |
| Rate limiting (token bucket) | `AddRateLimiter` (поверх `System.Threading.RateLimiting`) | `golang.org/x/time/rate`: `Allow`/`Wait`/`Reserve` |
| Таймаут | `AddTimeout` | `context.WithTimeout` ([Раздел 3](../03-concurrency/03-select-and-context.md)) |
| Fallback | `AddFallback` | `if err != nil { ... }` (язык, без библиотеки) |
| Хеджирование | `AddHedging` | `select` + несколько горутин (свой код) |
| Bulkhead / лимит конкурентности | `ConcurrencyLimiter` | буф. канал-семафор / `x/sync/semaphore` |
| Сборка нескольких паттернов | `ResiliencePipelineBuilder` (единый) | вложение вызовов функций руками |
| Сквозная отмена | `CancellationToken` в `ExecuteAsync` | `context.Context` первым аргументом |

## Плюсы и минусы каждого подхода

### Polly (единый фреймворк)

**Плюсы:**
- ✅ **Единый API и словарь** — один способ конфигурировать и исполнять все стратегии; новый паттерн добавляется тем же билдером.
- ✅ **Высокая открываемость** — есть одно место, одна документация, готовая интеграция с DI (`AddResiliencePipeline`), `HttpClientFactory` и телеметрией.
- ✅ **Богатый каталог из коробки** — hedging, fallback, bulkhead и пр. уже реализованы, протестированы и согласованы между собой.
- ✅ **Автоматическая композиция** — порядок «луковицы» задаётся декларативно порядком вызовов; меньше шансов собрать вложенность с ошибкой.

**Минусы:**
- ❌ **Больше абстракции и «магии»** — поведение пайплайна (что и в каком порядке оборачивает вызов) не всегда очевидно из места применения.
- ❌ **Вес и связанность** — тянете весь фреймворк и его модель, даже если нужен один простой ретрай.
- ❌ **Кривая обучения** — чтобы пользоваться грамотно, нужно знать модель Polly (стратегии, опции, предикаты, телеметрию).

### Модульный Go-подход (набор кирпичей)

**Плюсы:**
- ✅ **Минимализм** — для простого ретрая не нужна вообще никакая зависимость, хватит десяти строк цикла.
- ✅ **Явный контроль** — вы сами и в своём коде решаете порядок вложенности (retry внутри/снаружи breaker), видите каждую паузу и каждый токен; никакой скрытой механики.
- ✅ **Прозрачность и слабая связанность** — каждая библиотека крошечная, заменяема и понятна целиком; легко прочитать исходники.
- ✅ **Единый знаменатель — `context`** — все кирпичи уважают отмену/дедлайн, так что сквозная отмена работает «бесплатно».

**Минусы:**
- ❌ **Нет единого API** — три библиотеки = три разные философии и сигнатуры; склейку и согласованность держите вы.
- ❌ **Низкая открываемость** — нет одного «как у нас сделана отказоустойчивость»; знание размазано по библиотекам и принятым в команде соглашениям.
- ❌ **Ручная композиция = ручные ошибки** — неправильный порядок retry/breaker или забытый `ctx` в одном из слоёв не подсветит никакой фреймворк.
- ❌ **Каталог уже** — хеджирование, bulkhead, fallback придётся собирать самому из примитивов; готовых, «причёсанных» реализаций меньше.

> **Параллель с .NET:** это частный случай общего водораздела между экосистемами, который проходит через весь учебник. .NET тяготеет к мощным интегрированным фреймворкам (ASP.NET, EF Core, Polly), которые много берут на себя в обмен на изучение их модели. Go тяготеет к маленьким ортогональным библиотекам и явному коду «без магии», в обмен на то, что собирать целое вы будете руками. Ни один подход не «лучше» — это разные точки на оси «удобство и единообразие ↔ минимализм и контроль». Переходя на Go, не ищите «Polly для Go»: правильный вопрос — не «какой фреймворк всё сделает», а «какой кирпич нужен под конкретную задачу и как я вложу его в остальные».

## Итог

- Главное различие — **архитектурное, а не в возможностях**: .NET даёт **единый фреймворк** (Polly / `Microsoft.Extensions.Resilience`) с одним API, общим каталогом стратегий и автоматической композицией через `ResiliencePipeline`. Go даёт **набор независимых кирпичей** (`avast/retry-go`, `sony/gobreaker`, `x/time/rate`, плюс `context` для таймаутов и язык для fallback), которые вы складываете сами.
- Возможности сопоставимы почти один-в-один (см. таблицу), но в Go **timeout — это `context`, а fallback — обычный `if`**; отдельных примитивов под них не нужно.
- Polly выигрывает в **единообразии, открываемости и богатстве каталога** ценой абстракции и веса. Go выигрывает в **минимализме, прозрачности и явном контроле** ценой ручной склейки и отсутствия единого API.
- Объединяющий знаменатель всех Go-кирпичей — **`context.Context`**: уважение отмены и дедлайнов делает их совместимыми между собой и со всей экосистемой.
- Практический совет при переходе: не искать «Polly для Go», а подбирать узкий инструмент под задачу и осознанно собирать композицию (помня про порядок retry/breaker и сквозной `ctx`).

На этом раздел об отказоустойчивости завершён. Вы умеете повторять transient-сбои с backoff и jitter, отсекать упавшие зависимости предохранителем и держать частоту запросов в рамках — и понимаете, чем сборка этих паттернов в Go отличается от готового конвейера Polly.
