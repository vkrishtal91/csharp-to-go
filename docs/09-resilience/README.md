# Раздел 9. Resilience паттерны (Отказоустойчивость)

В распределённой системе отказы — это не исключительная ситуация, а норма. Сеть моргает, соседний сервис деплоится и на 5 секунд отдаёт `503`, база данных упирается в лимит соединений, внешний API вводит троттлинг. .NET-разработчик привык решать это одним инструментом: подключаешь **Polly**, оборачиваешь вызов в `ResiliencePipeline` — и получаешь ретраи, circuit breaker, таймауты, fallback из единого, хорошо документированного фреймворка. Это «батарейки в комплекте» от индустриального стандарта.

В Go подход прямо противоположный и отражает общую философию языка (см. [Раздел 1](../01-philosophy-and-syntax/README.md)): **нет единого фреймворка отказоустойчивости — есть набор маленьких, независимых решений, которые вы собираете сами.** Простой ретрай вы напишете циклом за десять строк. Где нужно больше — берёте узкоспециализированную библиотеку: `avast/retry-go` для ретраев, `sony/gobreaker` для предохранителя, `golang.org/x/time/rate` для лимитов. Каждая делает ровно одну вещь и не знает о других; склейка — ваша ответственность и ваш явный контроль.

Этот раздел проводит по трём фундаментальным паттернам — ретраи, circuit breaker, rate limiting — каждый сначала «руками», затем через идиоматичную библиотеку, и заканчивается прямым сравнением с экосистемой Polly. Сквозная нить здесь — `context.Context` (см. [Раздел 3](../03-concurrency/03-select-and-context.md)): любой механизм отказоустойчивости в Go обязан уважать отмену и дедлайны, иначе он сам становится источником зависаний.

## Цели обучения

После этого раздела вы будете:

- Понимать, **что можно ретраить, а что нельзя** (идемпотентность, transient-ошибки), и реализовывать ретраи с экспоненциальной задержкой (exponential backoff) и случайным разбросом (jitter), уважающие `context`.
- Уверенно применять `avast/retry-go`: `retry.Do`, опции `Attempts`/`Delay`/`DelayType`/`RetryIf`/`Context`.
- Объяснять паттерн **Circuit Breaker** и его три состояния (Closed → Open → Half-Open) и настраивать `sony/gobreaker` под реальную нагрузку.
- Понимать алгоритм **Token Bucket** и применять `golang.org/x/time/rate` (`Allow`/`Wait`/`Reserve`) для защиты себя и зависимостей.
- Грамотно **сочетать** паттерны (ретрай внутри/снаружи breaker, лимитер на входе) и понимать порядок их применения.
- Видеть архитектурную разницу: единый фреймворк Polly против модульного, децентрализованного подхода Go — и осознанно выбирать между «всё из коробки» и «минимализм + явный контроль».

## Ориентир: что в .NET → что в Go

Держите эту таблицу под рукой при чтении раздела. Детали каждого сопоставления разбираются в соответствующих главах.

| Задача / стратегия в .NET (Polly)              | Решение в Go                                    | Ключевое отличие                                                                 |
| ---------------------------------------------- | ----------------------------------------------- | -------------------------------------------------------------------------------- |
| `ResiliencePipeline` — единый конвейер стратегий | Собираете сами: композиция функций/обёрток       | Нет единого фреймворка; склейка паттернов — ваша явная ответственность           |
| `RetryStrategy` (`WaitAndRetry`, backoff, jitter) | Цикл `for` руками или `avast/retry-go`           | Простой ретрай — десять строк; библиотека для backoff/jitter/условий             |
| `CircuitBreakerStrategy`                       | `sony/gobreaker`                                | Отдельная библиотека на одну задачу                                              |
| `RateLimiterStrategy` (на базе `System.Threading.RateLimiting`) | `golang.org/x/time/rate`                         | В Go токен-бакет в `x/time` давно, в .NET встроенный лимитер только с .NET 7      |
| `TimeoutStrategy`                              | `context.WithTimeout` (см. [Раздел 3](../03-concurrency/03-select-and-context.md)) | Таймаут в Go — это часть `context`, а не отдельная «стратегия»                    |
| `FallbackStrategy`                             | Обычный `if err != nil { ... }`                  | Fallback в Go — это просто ветка обработки ошибки, не нужен отдельный примитив    |
| `HedgingStrategy` (хеджирование)               | `select` + несколько горутин руками               | Готового примитива нет; собирается из каналов и `select`                          |
| `Bulkhead` / `ConcurrencyLimiter` (изоляция)   | Буферизованный канал-семафор или `x/sync/semaphore` | Идиоматичный счётный семафор — буф. канал ([Раздел 3](../03-concurrency/README.md)) |
| `CancellationToken` сквозь весь пайплайн       | `context.Context` первым аргументом              | Один объект несёт отмену, дедлайн и значения                                      |
