# Раздел 1. Философия, Синтаксис и Структуры данных

Первое, что чувствует C#-разработчик, открыв Go: язык кажется почти пустым. Нет свойств, нет наследования классов, нет `async/await`, нет LINQ, нет тернарного оператора, нет перегрузки операторов, нет исключений. Где-то на третий день приходит понимание, что это не недоработка, а осознанный инженерный выбор. C# двадцать лет наращивал выразительность: каждая версия добавляла сахар (LINQ, `var`, `async`, pattern matching, records, top-level statements). Go двадцать лет почти ничего не добавлял и гордится этим.

Этот раздел закладывает фундамент. Мы разберём философию минимализма Go и почему «маленький язык» — это фича, а не баг. Затем — ключевую перенастройку мышления: в Go нет деления типов на «значимые» и «ссылочные», всё копируется по значению. На этой модели держатся структуры данных: массивы, слайсы и мапы, на которых в Go строится буквально всё (никакого богатого `System.Collections.Generic`). Отдельный, самый глубокий файл посвящён слайсам под капотом — это структура, которую обязан понимать каждый Go-разработчик, иначе вас ждут трудноуловимые баги с общей памятью. Завершаем разделом-мостиком, где собраны прямые сравнения конструкций C# и Go, включая «жизнь без LINQ».

## Цели обучения

После этого раздела вы будете:

- Понимать философию Go (ортогональность, «один способ сделать», единый форматтер) и почему она противоположна философии C#.
- Знать все 25 ключевых слов Go и встроенные функции (`builtin`), отличать их от предобъявленных идентификаторов (`nil`, `true`, `iota`).
- Понимать, что в Go нет категорий value-type/reference-type: всё копируется по значению, а «ссылочное» поведение — следствие указателя внутри значения; делиться данными вы решаете явно через указатель.
- Уверенно работать с массивами (value-семантика!), слайсами и мапами; реализовывать стек, очередь и множество (`HashSet`) на встроенных типах.
- Понимать внутреннее устройство слайса (хедер из трёх полей, `len`/`cap`, реаллокации при `append`) и избегать классических ловушек с общим backing-массивом.
- Переводить привычные LINQ-цепочки и коллекции `System.Collections.Generic` на идиоматичный Go.

## Ориентир: что в .NET → что в Go

| В .NET / C#                           | В Go                                   | Ключевое отличие                                                                   |
| ------------------------------------- | -------------------------------------- | ---------------------------------------------------------------------------------- |
| value type / reference type (категория типа) | Категории нет: всё копируется по значению | «Ссылочность» — следствие указателя внутри значения, а не свойство типа          |
| `class` vs `struct` решает автор типа | Значение `T` vs указатель `*T` в месте использования | Семантику выбираете вы на каждом параметре/присваивании, а не автор типа |
| LINQ (`Where`, `Select`, `Aggregate`) | Явные циклы `for` или дженерик-хелперы | Нет встроенного DSL над коллекциями; код многословнее, но прозрачнее по аллокациям |
| `List<T>` (reference type)            | Слайс `[]T`                            | Хедер слайса копируется по значению, но указывает на общий backing-массив          |
| `Dictionary<K,V>`                     | `map[K]V`                              | Порядок итерации намеренно случаен; `nil`-мапа читается, но паникует при записи    |
| `HashSet<T>`                          | `map[T]struct{}`                       | Готового типа нет; идиома — мапа со значением нулевого размера                     |
| `Stack<T>` / `Queue<T>`               | Слайс + `append`/срез                  | Готовых типов нет, реализуются в 1-2 строки на слайсе                              |
| `T[]` (массив — reference type)       | `[N]T` (массив — value type)           | Размер входит в тип; массив копируется целиком при присваивании и передаче         |
| `Span<T>` / `Memory<T>`               | Слайс `[]T`                            | Слайс изначально и есть «окно над памятью без копии»                               |
