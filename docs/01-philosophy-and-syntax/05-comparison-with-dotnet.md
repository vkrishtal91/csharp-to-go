# Сравнение с .NET

Этот файл — консолидированный мостик раздела. Мы собираем в одном месте прямые сопоставления конструкций C#/.NET и Go: что во что переводится, где семантика совпадает, а где радикально расходится. Если предыдущие файлы объясняли Go «изнутри», то здесь мы смотрим на него глазами .NET-инженера и отвечаем на вопрос «а как мне делать привычное X?».

Главная тема — **жизнь без LINQ**, потому что именно её отсутствие острее всего ощущает каждый, кто пришёл из C#. Дальше идут таблицы и развёрнутые сравнения по коллекциям, мапам, множествам, массивам и указатель на ближайший аналог слайсов в новом .NET — `Span<T>`.

## Богатый синтаксис C# против минимализма Go

Коротко напомним водораздел (подробно — в файле [Спартанский подход](./01-spartan-philosophy.md)). C# — язык-максималист: 100+ ключевых слов, несколько способов сделать почти всё, мощнейший вывод типов, метапрограммирование, сахар на каждом шагу. Go — язык-минималист: 25 ключевых слов, «один способ сделать», намеренно скромный синтаксис.

| Аспект | C# / .NET | Go |
|---|---|---|
| Ключевых слов | 100+ | 25 |
| Фильтрация/проекция коллекций | LINQ (DSL) | Явные циклы `for range` |
| Тернарный оператор | Есть (`?:`) | Нет |
| Циклы | `for`, `foreach`, `while`, `do-while` | Только `for` |
| Обработка ошибок | Исключения (`try/catch`) | Возврат `(value, error)` |
| Наследование | Классы, интерфейсы | Только интерфейсы + встраивание |
| Видимость | `public`/`private`/`protected`/`internal` | Регистр первой буквы |
| Форматирование кода | Настраиваемое (`.editorconfig`) | Жёсткий `gofmt`, без настроек |
| Дженерики | С 2005 (C# 2.0), очень богатые | С 2022 (Go 1.18), намеренно скромные |

Для .NET-разработчика это означает смену привычки: вместо «найду самый элегантный однострочник» — «напишу прямой, очевидный код». Поначалу это раздражает, потом становится понятно, что незнакомый Go-код читается без усилий именно потому, что в нём негде спрятать хитрость.

## Жизнь без LINQ

Это самая большая боль перехода. В C# вы привыкли писать декларативно:

```csharp
var names = users
    .Where(u => u.Age >= 18)
    .Select(u => u.Name)
    .OrderBy(n => n)
    .ToList();
```

В Go **нет LINQ** и нет встроенного DSL над коллекциями. И, что важнее, идиоматичный Go в большинстве случаев **не пытается его воспроизвести** — он предпочитает явные циклы. Это сознательный выбор: цикл прозрачен по аллокациям и поведению, его видно насквозь, тогда как ленивая цепочка LINQ скрывает, сколько раз и что аллоцируется.

### Where → цикл с условием

```csharp
var adults = users.Where(u => u.Age >= 18).ToList();
```

```go
var adults []User
for _, u := range users {
    if u.Age >= 18 {
        adults = append(adults, u)
    }
}
```

### Select → цикл с проекцией (с преаллокацией)

```csharp
var names = users.Select(u => u.Name).ToList();
```

```go
names := make([]string, 0, len(users)) // преаллокация, см. файл о слайсах
for _, u := range users {
    names = append(names, u.Name)
}
```

### Aggregate → цикл с аккумулятором

```csharp
var total = orders.Aggregate(0m, (sum, o) => sum + o.Amount);
```

```go
var total decimal // условно; в реальном Go взяли бы тип money/int-копейки
for _, o := range orders {
    total += o.Amount
}
```

### Дженерик-хелперы: можно, но дозированно

С появлением дженериков (Go 1.18+) ничто не мешает написать собственные `Map`/`Filter`/`Reduce`:

```go
func Map[T, R any](s []T, f func(T) R) []R {
    r := make([]R, len(s))
    for i, v := range s {
        r[i] = f(v)
    }
    return r
}

func Filter[T any](s []T, pred func(T) bool) []T {
    var r []T
    for _, v := range s {
        if pred(v) {
            r = append(r, v)
        }
    }
    return r
}
```

Существует и популярная библиотека **`github.com/samber/lo`**, которая даёт `lo.Map`, `lo.Filter`, `lo.Reduce`, `lo.GroupBy` и сотни других LINQ-подобных хелперов:

```go
import "github.com/samber/lo"

names := lo.Map(users, func(u User, _ int) string { return u.Name })
adults := lo.Filter(users, func(u User, _ int) bool { return u.Age >= 18 })
```

**Но честно: это спорный путь в Go-сообществе.** Идиоматичный Go чаще предпочитает явный цикл, и вот почему:

- **Читаемость.** `for ... range` с `if` понятен любому Go-разработчику мгновенно; цепочка из дженерик-хелперов с замыканиями читается тяжелее и непривычна для языка.
- **Производительность.** Каждый хелпер — это вызов функции и обычно отдельная аллокация промежуточного слайса. Цепочка `Map(Filter(...))` аллоцирует несколько временных слайсов; ручной цикл делает всё за один проход и одну аллокацию. В Go, где контроль над аллокациями — часть культуры, это весомый аргумент.
- **Отладка.** В цикл можно поставить точку останова и шагать; сквозь цепочку замыканий это менее удобно.

Практический совет: для простых одношаговых преобразований пишите цикл. Дженерик-хелперы или `lo` уместны, когда они **повышают** читаемость (например, явный `lo.GroupBy`), а не превращают код в нечитаемую цепочку. Не тащите LINQ-стиль в Go механически — это распространённая ошибка новичков из мира C#.

> **Параллель с .NET:** Важно понимать ещё и то, что LINQ ленив (отложенное выполнение через `IEnumerable<T>`), а Go-циклы и хелперы — нет, они выполняются немедленно и материализуют результат. Поэтому «бесплатные» в C# композиции (несколько `Where` подряд сливаются в один проход благодаря ленивости) в наивном Go-переводе превратятся в несколько реальных проходов с промежуточными слайсами. Ещё одна причина писать один явный цикл.

## List<T> против слайса

Внешне взаимозаменяемы, по семантике — разные. `List<T>` это reference type (объект-обёртка в куче), а слайс — это value type (хедер из трёх полей), который копируется при передаче, но указывает на общий backing-массив.

| Характеристика | `List<T>` (C#) | Слайс `[]T` (Go) |
|---|---|---|
| Природа | Reference type (объект в куче) | Value type (хедер из 3 слов) |
| Передача в метод | По ссылке: метод видит тот же объект | Хедер копируется; данные общие через указатель |
| Добавление | Метод `list.Add(x)` (мутирует объект) | `s = append(s, x)` (возвращает новый хедер) |
| Рост | Удвоение внутреннего массива | Реаллокация backing-массива (см. файл о слайсах) |
| Срез/подмножество | `GetRange` (копия) | `s[a:b]` (общая память!) |
| Ёмкость | `new List<T>(cap)` / `EnsureCapacity` | `make([]T, 0, cap)` |

**Самое каверзное — поведение при передаче в функцию.** Из-за того что хедер копируется, а данные нет, получается «полу-ссылочное» поведение:

```go
func mutate(s []int) {
    s[0] = 999      // ✅ видно снаружи: меняем общий backing-массив
    s = append(s, 1) // ❌ снаружи НЕ видно: append изменил локальную копию хедера
                     //    (а при реаллокации — ещё и отвязал её от исходного массива)
}

func main() {
    data := []int{1, 2, 3}
    mutate(data)
    fmt.Println(data) // [999 2 3] — изменение элемента видно, добавление — нет
}
```

```csharp
// C#: List<T> — всё видно, т.к. передаётся ссылка на один объект
void Mutate(List<int> list) {
    list[0] = 999; // видно снаружи
    list.Add(1);   // тоже видно снаружи!
}
```

Это ключевое отличие: с `List<T>` метод, добавляющий элемент, влияет на исходный список. Со слайсом — нет, потому что `append` меняет копию хедера. Чтобы «вернуть» рост наружу, в Go возвращают слайс из функции (`func grow(s []T) []T`) или передают указатель на слайс (`*[]T`).

## Dictionary<K,V> против map

Концептуально одно и то же (хеш-таблица), но есть три отличия, о которых нужно помнить.

| Характеристика | `Dictionary<K,V>` (C#) | `map[K]V` (Go) |
|---|---|---|
| Порядок итерации | Не гарантирован, но де-факто стабилен | Намеренно случаен от запуска к запуску |
| Нулевое/`null` состояние | `null` бросает NRE даже на чтение | `nil`-мапа: чтение ок, запись паникует |
| Отсутствие ключа | `KeyNotFoundException` на `[]`, либо `TryGetValue` | Возвращает нулевое значение; идиома `v, ok := m[k]` |
| Удаление | `dict.Remove(k)` | `delete(m, k)` |
| Потокобезопасность | `ConcurrentDictionary<K,V>` из коробки | Только `sync.Map` (нишевый) или ручной `sync.Mutex` |

Подробно `nil`-поведение и «comma ok» разобраны в файле [Коллекции](./03-collections.md). Здесь подчеркнём два пункта.

**Случайный порядок** — это намеренная защита от ошибочной зависимости от порядка (рантайм рандомизирует обход). В .NET порядок `Dictionary` был стабилен на практике, и код иногда неявно на это полагался; в Go так сделать невозможно. Нужен порядок — сортируйте ключи в отдельный слайс.

**Конкурентность.** В .NET есть готовый `ConcurrentDictionary<K,V>` — потокобезопасный и оптимизированный. В Go встроенная `map` **не** потокобезопасна, и одновременные чтение+запись из разных горутин вызывают аварию рантайма (он это детектирует и роняет программу с `concurrent map writes`). Готового аналога почти нет: есть `sync.Map`, но он узкоспециализирован (оптимизирован под специфические сценарии — «много чтений, ключи пишутся один раз») и потому применяется нечасто. В большинстве случаев конкурентный доступ к мапе в Go защищают вручную через `sync.Mutex`/`sync.RWMutex`. Тема конкурентности — это Раздел 3, здесь лишь фиксируем отсутствие готового `ConcurrentDictionary`.

## HashSet<T> против map[T]struct{}

Готового множества в Go нет — используют мапу с пустым значением. Полное объяснение (почему именно `struct{}` нулевого размера, идиома проверки) — в файле [Коллекции](./03-collections.md).

| Операция | `HashSet<T>` (C#) | `map[T]struct{}` (Go) |
|---|---|---|
| Создание | `new HashSet<T>()` | `make(map[T]struct{})` |
| Добавить | `set.Add(x)` | `set[x] = struct{}{}` |
| Проверить | `set.Contains(x)` | `_, ok := set[x]` |
| Удалить | `set.Remove(x)` | `delete(set, x)` |
| Размер | `set.Count` | `len(set)` |

Коротко: `HashSet<T>` в .NET сам построен поверх хеш-таблицы, так что `map[T]struct{}` — та же идея вручную, где `struct{}` (ноль байт) служит маркером «значение не нужно, важен ключ».

## Массивы: reference type в C# против value type в Go

Это одно из самых важных **концептуальных** отличий, на котором спотыкаются почти все.

```csharp
// C#: массив — REFERENCE type
int[] a = {1, 2, 3};
int[] b = a;       // b и a ссылаются на ОДИН массив
b[0] = 999;
Console.WriteLine(a[0]); // 999 — изменилось через обе ссылки
```

```go
// Go: массив — VALUE type
a := [3]int{1, 2, 3}
b := a            // b — ПОЛНАЯ КОПИЯ всех элементов
b[0] = 999
fmt.Println(a[0]) // 1 — оригинал не тронут
```

В C# массив — объект в куче, переменная хранит ссылку; присваивание копирует ссылку, не данные. В Go массив — это значение (как `struct`): присваивание, передача в функцию и возврат **копируют все элементы целиком**. Размер при этом ещё и часть типа (`[3]int ≠ [4]int`).

Именно из-за дороговизны копирования и негибкости размера в прикладном Go-коде почти везде используют **слайсы**, а не массивы. Слайс возвращает «ссылочное» (точнее, разделяемое через указатель) поведение, к которому вы привыкли с C#-массивами и `List<T>`. То есть: хотите «как массив в C#» — берите слайс, а не массив Go. Детали в файле [Коллекции](./03-collections.md).

## Span<T>/Memory<T> как ближайшая аналогия слайсам

Хорошая новость для тех, кто работал с современным .NET (Core/5+): у вас уже есть интуиция для слайсов — это `Span<T>` и `Memory<T>`. Все три выражают одну идею: **окно над памятью без копирования данных** (указатель + длина).

| Идея | `Span<T>` / `Memory<T>` (.NET) | Слайс `[]T` (Go) |
|---|---|---|
| Суть | Дескриптор (ptr + length) поверх чужой памяти | Хедер (ptr + len + cap) поверх backing-массива |
| Копирует данные при создании? | Нет — «вид» над существующим буфером | Нет — «вид» над backing-массивом |
| Срез | `span.Slice(a, b)` (общая память) | `s[a:b]` (общая память) |
| Рост (`append`) | Нет, размер фиксирован | Есть, через `append` (с возможной реаллокацией) |
| Ограничения размещения | `Span<T>` — `ref struct`: только на стеке, нельзя в поля/`async`/кучу | Нет ограничений: слайс живёт где угодно |
| Куда положить в кучу/поле | Для этого есть `Memory<T>` | Сам слайс это умеет «из коробки» |

Ключевые различия от `Span<T>`:

- **Слайс умеет расти.** `Span<T>` — это окно фиксированного размера; слайс через `append` может расширяться (с реаллокацией). По этому свойству слайс ближе к `List<T>`, чем к `Span<T>`.
- **У слайса нет ограничений размещения.** `Span<T>` — это `ref struct`, его нельзя хранить в полях обычного класса, использовать в `async`-методах или поднимать в кучу; именно для таких случаев в .NET ввели `Memory<T>`. Слайс свободно живёт в полях структур, передаётся в горутины, кладётся в мапы — никаких особых правил.

Так что ментальная модель: слайс — это как если бы `Span<T>` (окно над памятью), `List<T>` (умеет расти) и `Memory<T>` (можно хранить где угодно) слили в один-единственный встроенный тип. Понимая `Span<T>`, вы понимаете и «окно без копии» в основе слайса — остаётся добавить к этой картине рост через `append` и ловушки общего backing-массива из файла [Слайсы под капотом](./04-slices-internals.md).
