# Раздел 3. Конкурентность и Синхронизация

В .NET конкурентность — это во многом библиотечная история. Да, у вас есть `Thread` и `ThreadPool` в рантайме, но вся эргономика асинхронного кода (`async`/`await`, `Task`, TPL, TPL Dataflow, `System.Threading.Channels`) живёт в библиотеках поверх CLR и развивалась годами. В Go конкурентность — это встроенная фича самого языка. Ключевое слово `go`, каналы (`chan`), оператор `select`, планировщик горутин — всё это часть спецификации языка и рантайма, а не отдельный фреймворк.

Этот сдвиг меняет стиль кода радикально. Вместо «заражающего» сигнатуры `async` и явных `await` вы пишете обычный, на вид синхронный, последовательный код — а рантайм Go незаметно превращает блокирующие вызовы в кооперативные точки переключения. Вместо `Task<T>` как контейнера будущего результата вы запускаете горутину и забираете результат через канал. Философию языка лаконично формулирует знаменитый девиз сообщества Go:

> **Do not communicate by sharing memory; instead, share memory by communicating.**
> *(Не общайтесь через разделяемую память — разделяйте память через общение.)*

То есть предпочтительный способ координации — не блокировки вокруг общих данных, а передача данных по каналам. Блокировки (`sync.Mutex` и компания) в Go тоже есть и активно используются, но это «второй выбор» для конкретных сценариев, а не основной инструмент.

## Чему вы научитесь

- Понимать, что такое горутина и почему она дешевле потока ОС на три порядка.
- Разбираться в модели планировщика **GMP** (Goroutine — Machine — Processor): work-stealing, вытеснение, обработка блокирующих syscalls и сетевого I/O.
- Использовать каналы как примитив коммуникации: буферизованные и небуферизованные, направленные, закрытие, типичные паттерны (pipeline, fan-in/fan-out).
- Мультиплексировать операции через `select`, делать таймауты и строить отмену дерева горутин через `context.Context`.
- Применять `sync.Mutex`, `sync.RWMutex`, `sync.WaitGroup`, `sync/atomic` там, где каналы избыточны, и не попадаться на нереентерабельность мьютекса.
- Находить и предотвращать **goroutine leaks** — главную «утечку» в конкурентном Go-коде — и ловить гонки детектором `-race`.
- Понимать жизненный цикл горутины (как она завершается и освобождает память) и дисциплину гарантированного завершения; правильно закрывать каналы, HTTP-body и стримы.

## Таблица-ориентир: .NET → Go

Держите её под рукой при чтении раздела. Детали и нюансы каждого сопоставления разбираются в соответствующих главах.

| Концепция в .NET | Аналог в Go | Ключевое различие |
| --- | --- | --- |
| `Task` / `Task.Run` / `async`-метод | горутина (`go f()`) | Горутина ничего не возвращает; результат — через канал |
| `async`/`await`, «цвет функции» | обычный синхронный код | В Go нет «цвета»: блокировка дёшева, сигнатуры не заражаются |
| `ConfigureAwait(false)` | не существует | Нет `SynchronizationContext` — нечего конфигурировать |
| `CancellationToken` / `CancellationTokenSource` | `context.Context` / `context.WithCancel` | Context несёт ещё дедлайны и значения единым объектом |
| `lock` (Monitor) | `sync.Mutex` | `lock` реентерабельный, `sync.Mutex` — **нет** |
| `ReaderWriterLockSlim` | `sync.RWMutex` | — |
| `SemaphoreSlim` | буферизованный канал или `golang.org/x/sync/semaphore` | Буф. канал — идиоматичный счётный семафор |
| `Interlocked` | пакет `sync/atomic` | — |
| `Task.WaitAll` / `Task.WhenAll` | `sync.WaitGroup` | WaitGroup ждёт горутины, у которых нет «результата» |
| `Task.WhenAny` / `Task.Delay` | `select` + `time.After` | `select` — оператор языка, а не метод |
| `System.Threading.Channels` (`Channel<T>`) | `chan T` | `chan` — на уровне языка, работает с `select` |
| TPL Dataflow (`BufferBlock<T>` и т.п.) | каналы + горутины (pipeline) | В Go конвейеры собираются вручную из примитивов |
| `ThreadPool` + планировщик задач | планировщик GMP в рантайме | Оба с work-stealing, но горутина ≠ Task ≠ поток |
| `LazyInitializer` / `Lazy<T>` (однократная инициализация) | `sync.Once` | — |
