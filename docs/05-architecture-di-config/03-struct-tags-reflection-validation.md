# Тэги структур, рефлексия, валидация

В двух предыдущих главах тэги вроде `env:"ADDR"` и `validate:"required"` уже мелькали. Пора разобрать механизм под ними. В .NET роль «метаданных на члене типа» играют **атрибуты** (`[Required]`, `[JsonPropertyName("id")]`, `[Range(1, 100)]`) — типизированные классы, которые рантайм и библиотеки читают через рефлексию. В Go этой роли служат **struct tags** — но устроены они принципиально проще и, как мы увидим, менее безопасно.

## Struct tags: что это и синтаксис

Struct tag — это **строковый литерал**, приписанный к полю структуры в обратных кавычках (backtick). Формально это просто строка-метаданные, которую сам компилятор Go никак не интерпретирует — он лишь сохраняет её и делает доступной через рефлексию. Смысл строке придают **библиотеки**, которые её читают.

Конвенция формата (её соблюдает стандартная библиотека и почти вся экосистема) — это набор пар `ключ:"значение"` через пробел:

```go
type User struct {
	ID    int    `json:"id"`
	Email string `json:"email" validate:"required,email"`
	Name  string `json:"name,omitempty" validate:"min=3,max=50"`
	Token string `json:"-"` // "-" у encoding/json означает «не сериализовать»
}
```

Разберём синтаксис на поле `Name`:

- Внутри backtick — несколько **тэгов через пробел**: `json:"..."` и `validate:"..."`. Каждый адресован своей библиотеке (по ключу).
- Значение каждого тэга — в **двойных кавычках**.
- Внутри значения — **опции через запятую**: у `json` это имя поля `name` плюс опция `omitempty` (не выводить пустое значение). У `validate` — список правил `min=3,max=50`.

Ключ перед двоеточием — это «адрес»: библиотека `encoding/json` читает только ключ `json`, валидатор — только `validate`, ORM — свой (`gorm`/`db`). Одно поле может нести тэги для скольких угодно потребителей одновременно.

**Кто использует struct tags:**

| Ключ тэга | Потребитель | Назначение |
| --- | --- | --- |
| `json` | `encoding/json` | Имена полей и опции при JSON-сериализации (Раздел 6) |
| `yaml` | `gopkg.in/yaml.v3` | То же для YAML |
| `env`, `envDefault` | `caarlos0/env`, `cleanenv` | Маппинг переменных окружения (глава 2) |
| `validate` | `go-playground/validator` | Правила валидации (ниже) |
| `db` | `sqlx`, ORM | Соответствие колонкам БД (Раздел 10) |

> **Параллель с .NET:** struct tag ≈ атрибут на свойстве. `json:"id"` ≈ `[JsonPropertyName("id")]`, `validate:"required"` ≈ `[Required]`, `db:"user_id"` ≈ `[Column("user_id")]`. Но есть фундаментальная разница в безопасности: **атрибут — это типизированный класс**, его аргументы проверяет компилятор, а опечатка в имени даст ошибку сборки. **Struct tag — это просто строка.** Опечатку `valdate:"required"` или `validate:"requried"` компилятор **не заметит** ❌ — тэг будет молча проигнорирован, и валидация не сработает. Это плата за простоту механизма; ловится она линтерами и тестами, а не типобезопасностью.

## Рефлексия: как тэги вообще читаются

Чтобы из строки-тэга получилась польза, кто-то на рантайме должен пройтись по полям типа и эти строки достать. Это делает пакет [`reflect`](https://pkg.go.dev/reflect) — механизм интроспекции типов и значений во время выполнения, прямой аналог `System.Reflection`.

Две точки входа:

- `reflect.TypeOf(x)` — даёт `reflect.Type`: метаданные **типа** (имена полей, их типы, тэги). Аналог `typeof(T)` / `x.GetType()`.
- `reflect.ValueOf(x)` — даёт `reflect.Value`: обёртку над **значением**, через которую можно читать и (с оговорками) писать поля. Аналог работы с `PropertyInfo.GetValue/SetValue`.

Вот как библиотека вроде `caarlos0/env` принципиально читает тэг `env` (упрощённо):

```go
func describe(v any) {
	t := reflect.TypeOf(v)
	if t.Kind() == reflect.Pointer {
		t = t.Elem() // разыменуем, если передали указатель
	}
	for i := 0; i < t.NumField(); i++ {
		field := t.Field(i)                       // reflect.StructField
		envName := field.Tag.Get("env")           // достаём значение тэга по ключу
		fmt.Printf("поле %s ← переменная %q\n", field.Name, envName)
	}
}
```

`field.Tag` имеет тип `reflect.StructTag` (та самая строка), а `.Get("env")` парсит её по конвенции и возвращает значение нужного ключа. Именно так конфиг-библиотеки сопоставляют поля с переменными окружения, `encoding/json` — с ключами JSON, а валидатор — с правилами. Без рефлексии тэги были бы мёртвым грузом: строка, которую некому прочитать.

> **Параллель с .NET:** `reflect.TypeOf` ≈ `typeof`/`GetType`, `field.Tag.Get("env")` ≈ `prop.GetCustomAttribute<EnvAttribute>()`. Концептуально одно и то же: на рантайме обойти члены типа и прочитать их метаданные. Разница — в форме метаданных (типизированный атрибут vs строковый тэг) и в том, что в Go рефлексия по культуре считается «тяжёлым инструментом на крайний случай», тогда как в .NET она встроена в повседневные фреймворки глубже и привычнее.

## Цена рефлексии и почему её дозируют

Рефлексия в Go работает, но она **заметно медленнее** прямого кода и обычно вызывает **аллокации** (упаковка значений в `interface{}`, см. [Раздел 2](../02-memory-gc-and-types/05-any-boxing-and-generics.md)). Прямое обращение `cfg.Addr` компилятор превращает в пару инструкций; добывание того же поля через `reflect.Value.Field(i).String()` — это поиск по метаданным, проверки и боксинг на каждом шаге.

Практический вывод, который определяет стиль Go-кода:

- ✅ Рефлексия **уместна там, где она исполняется редко или вне горячего пути**: разбор конфига на старте, биндинг/валидация входящего запроса, (де)сериализация в `encoding/json`. Здесь её цена незаметна на фоне сети и I/O.
- ❌ Рефлексию **избегают в горячих циклах** и на путях с высокой нагрузкой — там, где те же микросекунды множатся на миллионы вызовов.

И ключевое культурное отличие от .NET: вместо рефлексии Go-сообщество часто выбирает **кодогенерацию** — сгенерировать обычный, статически типизированный Go-код на этапе сборки (тема [Раздела 7](../07-code-generation/README.md)). Там, где .NET положился бы на Reflection Emit на рантайме, Go предпочтёт `go generate` и готовый код, который видит компилятор. Так из самого языка вытесняется «магия рантайма»: быстрее, проще для отладки, ошибки ловятся при сборке.

> **Параллель с .NET:** в .NET у вас тоже есть выбор «рефлексия vs Source Generators»: `System.Text.Json` умеет и рефлексивный режим, и режим с source-generated контекстом для AOT/скорости. Go делает тот же выбор более радикально и по умолчанию в пользу кодогенерации, потому что рефлексия здесь дороже относительно остального языка и противоречит его установке на явность.

## Валидация: `go-playground/validator`

Самый частый «потребитель» тэгов после сериализации — валидация. Стандартный в экосистеме инструмент — [`go-playground/validator/v10`](https://github.com/go-playground/validator) (актуальная ветка v10.x). Он читает тэг `validate` через рефлексию и проверяет поля по списку правил.

```go
package main

import (
	"fmt"

	"github.com/go-playground/validator/v10"
)

type SignupRequest struct {
	Email    string `validate:"required,email"`
	Username string `validate:"required,min=3,max=20,alphanum"`
	Age      int    `validate:"gte=18,lte=120"`            // 18 ≤ Age ≤ 120
	Password string `validate:"required,min=8"`
	Website  string `validate:"omitempty,url"`             // если задан — должен быть URL
	Role     string `validate:"oneof=admin user guest"`   // только из списка
}

func main() {
	// Валидатор переиспользуют (он кэширует разбор тэгов по типам).
	validate := validator.New(validator.WithRequiredStructEnabled())

	req := SignupRequest{Email: "not-an-email", Username: "ab", Age: 15, Password: "short"}

	if err := validate.Struct(req); err != nil {
		// Ошибки приходят как набор по каждому нарушенному полю.
		var verrs validator.ValidationErrors
		if errors.As(err, &verrs) {
			for _, fe := range verrs {
				fmt.Printf("поле %q нарушило правило %q (значение: %v)\n",
					fe.Field(), fe.Tag(), fe.Value())
			}
		}
		return
	}
	fmt.Println("валидно ✅")
}
```

Та же модель в .NET — это атрибуты DataAnnotations на свойствах:

```csharp
// .NET: правила — типизированные атрибуты; опечатку в них поймает компилятор.
public sealed class SignupRequest
{
    [Required, EmailAddress]
    public string Email { get; init; } = "";

    [Required, StringLength(20, MinimumLength = 3)]
    public string Username { get; init; } = "";

    [Range(18, 120)]
    public int Age { get; init; }
}
// var ok = Validator.TryValidateObject(req, ctx, results, validateAllProperties: true);
```

Что важно в Go-варианте:

- `validate.Struct(req)` через рефлексию обходит поля, читает их тэги `validate` и применяет правила. Это типичный «нечастый» путь (валидация входящего запроса), где цена рефлексии оправдана.
- Распространённые правила: `required`, `email`, `url`, `min`/`max` (длина строки/слайса или граница числа в зависимости от типа), `gte`/`lte`, `oneof`, `alphanum`, `uuid`. `omitempty` означает «пустое значение пропускаем, но если задано — проверяем».
- Ошибка приводится к `validator.ValidationErrors` (срез) — по элементу на каждое нарушение, с именем поля и сработавшим правилом. Удобно для формирования ответа API.
- Объект `validator.New(...)` **переиспользуют** (один на приложение): он кэширует разбор тэгов по типам, амортизируя рефлексию.

### Кастомные правила

Когда встроенных правил мало, регистрируют свою функцию и ссылаются на неё в тэге по имени:

```go
// Своя проверка: строка не должна содержать пробелов.
validate.RegisterValidation("nospaces", func(fl validator.FieldLevel) bool {
	return !strings.Contains(fl.Field().String(), " ")
})

type Account struct {
	Slug string `validate:"required,nospaces"` // ссылаемся на кастомное правило по имени
}
```

Функция получает `validator.FieldLevel` (доступ к значению поля и контексту) и возвращает `bool`. Этого достаточно для большинства доменных проверок; для межполевых правил есть `RegisterStructValidation`.

> **Параллель с .NET:** `validate:"required,email"` ≈ навесить `[Required]` и `[EmailAddress]` на свойство; `validate.Struct(req)` ≈ `Validator.TryValidateObject(...)` (или автоматическая валидация модели в ASP.NET Core); `RegisterValidation("nospaces", ...)` ≈ кастомный `ValidationAttribute` с переопределённым `IsValid`. Поведение похоже, но повторим главное различие: правила в C# — это **типизированные атрибуты** (опечатку в имени поймает компилятор), а в Go — **строки в тэге** (`validate:"requried"` молча не сработает ❌). Поэтому в Go-проектах валидацию принято **покрывать тестами** плотнее: тест на «невалидный ввод действительно отвергается» заодно ловит и опечатки в тэгах, которых не видит компилятор.

## Итог

- **Struct tag** — это строка-метаданные в backtick на поле, формата `ключ:"значение,опции"`; компилятор её не трактует, смысл придают библиотеки, читающие свой ключ (`json`, `env`, `validate`, `db`).
- Тэги достаёт **рефлексия** (`reflect`): `TypeOf`/`ValueOf`, `field.Tag.Get("ключ")` — прямой аналог чтения атрибутов через `System.Reflection`.
- Главное отличие от .NET-атрибутов — **безопасность**: атрибут типизирован и проверяется компилятором, а тэг это строка, и **опечатку в нём компилятор не ловит** ❌. Спасают линтеры и тесты.
- **Цена рефлексии** ощутима (медленнее + аллокации), поэтому в Go её применяют **дозированно**: на старте, при биндинге/валидации/сериализации — да; в горячих циклах — нет. Альтернатива по культуре — **кодогенерация** (Раздел 7) вместо рантайм-магии.
- **Валидация** — `go-playground/validator/v10`: правила в тэге `validate:"required,email,min=3"`, `validate.Struct(...)` обходит поля рефлексией, ошибки приходят как `ValidationErrors`; кастомные правила регистрируются по имени. Валидатор переиспользуют (кэш тэгов).
